## Summary

This PR addresses a real bug surfaced by the ClickUp ticket: the Lab Test Mapping form's "Lab Test" `<Select>` was returning tests that were already attached to a `LabGroup`, which let a user map the same test into two groups and duplicate its `LabTemplateItem` / `LabTestMapping` rows. The fix is two-pronged: (1) the repository's `buildWhereQuery` now exposes an `excludeLabTestsUsedInTemplates` flag that adds a `where.NOT = { labTemplateItems: { some: {} } }` clause (i.e. "hide any lab test that is referenced by at least one template item row"), and (2) the form now also passes `status: "ACTIVE"` so the dropdown is restricted to active tests, matching the rest of the lab mapping UX.

The change is small and the core intent is correct — but there are two non-trivial correctness problems and one cleanliness issue that should be addressed before merge: (a) the **old `checkExistInLabTemplate` flag is now dead code** that still parses in the Zod schema and silently no-ops on the `templateId` branch of `buildWhereQuery`, leaving an inconsistent state (a query param that is honored or ignored depending on the caller); (b) the new filter is too aggressive — it excludes *any* lab test that is referenced by *any* template item, but the actual ticket is "Prevent Duplicate Allocation **and Grouping**", which means the right filter is "hide tests already mapped to *this* `labGroup` (or to a `LabGroup` via `labTestMappings`)", not "hide tests that already have template items" (a lab test with template items but no group is still a valid candidate for grouping, and a lab test with template items can be safely mapped to a group — the duplicate comes from a second `labTestMappings` row, not from the template-item existence); and (c) the form change silently couples the lab-test dropdown to "ACTIVE only" — if a user opens the edit dialog for an `INACTIVE` lab group whose mapped test is itself `INACTIVE`, the test is now missing from the list with no warning.

The PR has no tests, no migration (none needed), and no permission changes — the lab-test-mapping form already sits behind the existing lab permission module.

## Issues

### Critical

- **`lab-test.repository.ts:192-198` — Filter logic solves the wrong problem**
  The ClickUp ticket title is "Prevent Duplicate Allocation and Grouping of Lab Test in Lab Test Mapping". The current `excludeLabTestsUsedInTemplates` filter excludes *any* lab test that has at least one `LabTemplateItem` row, regardless of whether that template item is part of a `LabGroup` / `LabTestMapping`. A lab test that is part of *zero* template items but is *already mapped to a `LabGroup`* via `labTestMappings` will still be selectable — and the user can add it to *another* group, which is the exact duplicate the ticket is complaining about. Conversely, a lab test that already has template items but is *not yet grouped* is hidden from the dropdown, which blocks legitimate work (the user may want to group an ungrouped test that happens to have template items).

  The model relations (from `prisma/schema.prisma:2935` and `:2936`):
  - `LabTest.labTestMappings LabGroup[]` — many-to-many to `LabGroup` (this is the "grouping" relation)
  - `LabTest.labTemplateItems LabTemplateItem[]` — one-to-many to `LabTemplateItem` (this is the template relation, NOT the grouping)

  The right filter is to exclude tests that already appear in `labTestMappings`, optionally restricted to the current edit target. For example:
  ```ts
  where.NOT = {
    labTestMappings: {
      some: query.templateId
        ? { labGroupId: { not: query.templateId } }
        : {},
    },
  };
  ```
  This is also what the **commented-out** block at `lab-test.repository.ts:145-178` was trying to do (it scopes by `query.templateId` via `labTemplateCategory.labTemplateId`). The new block loses that scoping and changes the semantic meaning.

  Evidence: `lab-test.repository.ts:192-198` — `where.NOT = { labTemplateItems: { some: {} } };` reads as "hide any test with any template item row", not "hide any test already in another group".

### High

- **`lab-test.repository.ts:180-190` — Dead `checkExistInLabTemplate` branch becomes a silent no-op**
  After this PR the only caller of the `checkExistInLabTemplate` flag is gone (the form switched to `excludeLabTestsUsedInTemplates`), but the schema (`lab-test-query.schema.ts:19`), the repository (`lab-test.repository.ts:180-190`), and the commented-out legacy block (`lab-test.repository.ts:145-178`) all still exist. Worse: the remaining `if (query.checkExistInLabTemplate && query.templateId)` branch now silently no-ops when the caller passes `checkExistInLabTemplate: true` *without* `templateId` (it used to fall through to the commented-out `some: {}` behavior). Future callers that pass the old flag will get a misleadingly empty result set instead of an error.

  Fix: delete the dead branch and the commented-out block, drop `checkExistInLabTemplate` from the schema. If the old flag is intentionally retained for API back-compat (a public query param), at minimum guard it with a `console.warn` or reject it via Zod `.refine()` so callers learn.

  Evidence: `lab-test-query.schema.ts:19` — `checkExistInLabTemplate: z.coerce.boolean().optional(),` still declared; `lab-test.repository.ts:180` — `if (query.checkExistInLabTemplate && query.templateId)` now never fires because no caller passes `templateId`; the only consumer, `lab-test-mapping-form.tsx:88`, just stopped sending the flag.

- **`lab-test-mapping-form.tsx:88` — Edit mode now hides the currently mapped test**
  The form is reused for both create (`isEdit = !!labGroup` → false) and edit (`isEdit = true`). On edit, the form's `labTestId` default value comes from `labGroup?.id` (`lab-test-mapping-form.tsx:56`), and the `<Select>` is driven by `labTestOpts` (the same `makeGetLabTest` query). After this PR, that query filters to `status: "ACTIVE"` AND `excludeLabTestsUsedInTemplates: true`. If a user clicks "Edit" on an `INACTIVE` lab group, or on a group whose mapped test has been `INACTIVE`d, the test vanishes from the dropdown — the field renders empty even though the form's internal state still has the ID set. Sample-type and collection-method text inputs continue to show stale data because `labTestData` is fetched by id directly, but the dropdown is desynchronized from the form state.

  Fix: either (a) add the currently-mapped test id to a `where.OR` / bypass list when `isEdit` is true (pass `includeLabTestId: labGroup.id` to the query, since `lab-test.repository.ts:141-143` already honors `query.includeLabTestId`), or (b) only pass `status: "ACTIVE"` and `excludeLabTestsUsedInTemplates: true` when `!isEdit`. (a) is cleaner — the schema already supports it.

  Evidence: `lab-test-mapping-form.tsx:83-90` — the same `makeGetLabTest` query powers both create and edit; `lab-test.repository.ts:141-143` — the `includeLabTestId` override branch exists but isn't used here.

### Medium

- **`lab-test-mapping-form.tsx:84-90` — Combined flags couple unrelated concerns and overload one query**
  The query key is now `["lab-tests", { page: 1, limit: 0, offset: 0, excludeLabTestsUsedInTemplates: true, status: "ACTIVE" }]`. Two flags were added that change two unrelated semantics: (1) hide tests already in templates, and (2) hide inactive tests. Bundling them in one flag makes it impossible for another caller to ask for "active only, no template filter" or "exclude duplicates, include inactive". Each is its own concern and should be its own optional query parameter. Concretely, drop the new `excludeLabTestsUsedInTemplates` flag from this call and add a separate `isActive: true` query parameter (the repository's `where.isActive` branch already exists at `lab-test.repository.ts:208-211`); or split `excludeLabTestsUsedInTemplates` out of the form's query and pass it explicitly via the new flag.

  Evidence: `lab-test-mapping-form.tsx:84-90` — single `makeGetLabTest` call with three independent filters collapsed into two flags.

- **No tests**
  The repository change is exactly the kind of Prisma `where` mutation that needs a Jest test for the create flow (no duplicates appear in the dropdown after the filter), the edit flow (currently-mapped test still appears), and the ungrouped-but-template-using case (still selectable). `lab-test.repository.ts` has no test file today (`find` returned no `*.test.ts` for it), but the schema enum change is non-trivial. Add `__tests__/lab-test.repository.test.ts` covering the three scenarios.

### Low / Nit

The Prisma schema has a corresponding `Status` enum (or at least a documented string union — verify with the team); using `z.enum(["ACTIVE", "INACTIVE"])` would give the same runtime semantics with proper type inference and a clearer error message on invalid input. Trivial change, but worth doing while touching this file.
