## Issues

### Low

- **`src/app/(dashboard)/hd/features/components/hd-bill-info-card.tsx:70-79` — Repeated optional chain `data?.ipdDailyBill?.admission`**
  The new block reads the same relation three times — once in the guard and twice inside the JSX:
  ```tsx
  {data?.ipdDailyBill?.admission && (
    <div>
      <Text size="sm" fw={500}>
        Admission ID:&nbsp;
        <Text span fw={400}>
          {data?.ipdDailyBill?.admission?.admissionId}
        </Text>
      </Text>
    </div>
  )}
  ```
  Strictly speaking the second `data?.ipdDailyBill?.admission` could be a non-null assertion (`data.ipdDailyBill!.admission!.admissionId`) because the guard proves the path is non-null, but that would introduce a non-null `!` (a known gotcha in this project, see `CLAUDE.md` "Known HMS gotchas"). The safer refactor is to lift the value into a local:
  ```tsx
  const admission = data?.ipdDailyBill?.admission;
  ...
  {admission && (
    <div>
      <Text size="sm" fw={500}>
        Admission ID:&nbsp;
        <Text span fw={400}>{admission.admissionId}</Text>
      </Text>
    </div>
  )}
  ```
  This also lets a future reviewer add `admission.discharges?.isFinalBillPaid` (which already exists on the type at `proxy-bill.type.ts:97-99`) without repeating the chain. Not a blocker — the existing `appointment` block at `:60-69` has the same repetition.

### Nit

- **`src/app/(dashboard)/hd/features/components/hd-bill-info-card.tsx:73-78` — `&nbsp;` separator instead of CSS gap**
  The new block uses `Admission ID:&nbsp;` to space label and value, matching the immediately-prior `Appointment ID` block at `:63`. The codebase is split on this — other label/value rows in the same file (e.g. `:19-23`, `:26-28`, `:31-34`) use a plain space and rely on the wrapping `<Text>`'s natural whitespace, which works because the colon + closing `</Text>` is followed by a literal space before the `<Text span>`. The `&nbsp;` form prevents wrapping the label to the next line and is the right call here. Keep as-is — flagging only because the pattern is inconsistent within the same component. Purely cosmetic.
- **`src/app/(dashboard)/hd/features/components/hd-bill-info-card.tsx:16` — `SimpleGrid cols={4}` will now produce a 4th-column shift**
  With the addition of this row (and the appointment row, which was already conditional), the grid renders 13 cells when both `appointment` and `admission` are present, 12 when only one is, 11 when neither is. Mantine `SimpleGrid cols={4}` wraps cleanly, so this is not a bug — but if the design intent was "always fill the row", the new row belongs in the always-shown section rather than as a conditional. Confirm with the design that the orphan-cell layout is acceptable when only one of `appointment` / `admission` is set. Purely UX.
