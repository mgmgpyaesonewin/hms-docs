## Summary

This PR addresses a real and reproducible category of bug in the lab-template feature: the gender/age-range conflict detector in `lab-template-category-services.tsx`, `lab-template-microbiology-template-form.tsx`, and the `findMatchingStatusName` matcher in `general-utils.ts` did not treat the `",-,"` / `"0,-,0"` / `""` empty-pattern range as "ALL-age" when comparing against non-empty ranges. The previous code blocked ALL-age entries when *any* MALE/FEMALE row existed (regardless of age overlap) and blocked ALL-age ALL-gender from coexisting with a specific-range MALE/FEMALE entry even when the ranges didn't overlap. The PR rewrites both conflict helpers to use a shared `doAgeRangesOverlap` (now living in `general-utils.ts:611`) and a small `isAllAgeRange` / `isEmptyOrDefault` helper, deduplicates the logic across the two forms, and also fixes a stale-state / missing-error-path bug in `useTemplateActions` in the add-service form.

The changes are mostly correct in *intent* and the helper-function extraction is the right move. But the PR has one critical correctness regression in `get-lab-template.api.ts:22` (removing `query` from the React Query key) that will cause cross-filter cache poisoning in the lab-template list, several `console.error` / `console.log` calls left in production code paths, a duplication smell where the same `isEmptyOrDefault` + `doAgeRangesOverlap` pair is now copy-pasted into 3 files (one of them without importing the new helper), and a few smaller issues (silent catch around the server action, weakened `isAllAgeRange` semantics, an asymmetric overlap case where `null` is parsed but `null` is also a valid input).

## Issues

### Critical

- **`src/app/(dashboard)/lab/lab-template/features/api/get-lab-template.api.ts:22` — Removing `query` from the queryKey causes cross-filter cache poisoning**
  Before: `queryKey: ["lab-template", query]`. After: `queryKey: ["lab-template"]`. TanStack Query's default key matcher is **exact**, not hierarchical — two calls with different `query` arguments now share a single cache entry. The first call (e.g. `query = { status: "ACTIVE" }`) caches the response, and the next call (e.g. `query = { status: "INACTIVE", page: 2 }`) reads the stale cached response. The user sees the wrong filter results on screen after navigating between filters. Additionally, every `queryClient.invalidateQueries({ queryKey: ["lab-template"] })` call in the codebase (lines 858, 52 of `delete-lab-template-category-modal.tsx`, line 2237 of `lab-template-microbiology-template-form.tsx`) now invalidates *all* cached variants — which sounds like an improvement but means every CRUD operation triggers a refetch for every filter combination, killing tab-state locality. **Fix:** restore the array form: `queryKey: ["lab-template", query] ?? {}]` or, if the goal was to make invalidation easier, move to the documented TanStack Query pattern `queryKey: ["lab-template", query ?? "all"]` with `exact: false` on the invalidation calls. There is no comment explaining the change, which suggests it was an unintended regression — `git blame` on this line will tell.
  Evidence: `get-lab-template.api.ts:22` — `queryKey: ["lab-template"]` (no second element); the old line was `queryKey: ["lab-template", query]`.

  This is the *other* side of the Critical issue: when the queryKey was `["lab-template", query]`, an `invalidateQueries({ queryKey: ["lab-template"] })` call (without `exact`) matched the prefix and invalidated every cached filter variant. After this PR changes the key to `["lab-template"]`, those invalidations *still* match (the prefix matches by definition now), but TanStack Query's behaviour on prefix-match invalidation differs subtly from exact-key invalidation: the matching `query` argument becomes irrelevant, so every variant is invalidated together. The PR needs to revert this single-element key (see Critical issue) **and** keep the prefix-match invalidations — those are correct for the old key. Do not "fix" this by adding `exact: true` to the invalidation calls; that breaks the prefix-match behavior the codebase relies on.
