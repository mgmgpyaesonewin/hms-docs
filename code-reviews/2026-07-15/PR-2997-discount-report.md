## Summary

Title says "Discount Report" but this is the umbrella PR that lands the **entire fee-report suite** end-to-end: nine materialized report tables (`cf_fee_reports`, `tc_fee_reports`, `ihd_fee_reports`, `pf_fee_reports`, `rf_fee_reports`, `reading_fee_reports`, `opd_income_reports`, `discount_reports`, plus `event_outbox`), 10 migrations, a shared `fee-report-shared/` component layer consumed by six report pages, six new API route groups that proxy to a separate `summary-api` microservice, six new event-emit helpers (`opd-billing-events`, `ipd-daily-bill-events`, `ipd-final-bill-events`, `lab-reading-events`, `imaging-reading-events`), and outbox-emit hooks wired into ~13 service/repository files. The wiring implements the transactional-outbox pattern from CLAUDE.md: source writes append an `event_outbox` row in the same Postgres tx, and a downstream `summary-service` worker materializes the report rows.

Strengths: events correctly use the caller's transaction, all `enqueue*` helpers no-op cleanly when `DEFAULT_TENANT_ID` is unset, payload discipline is consistent (minimal, worker re-reads the DB), and the `next.config.ts` `generateRouteRewrites` rule already maps `/reports/*` to `common/reports/*` so the sidebar hrefs work despite the route-group path mismatch.

Weaknesses: bulk pay/revert across all six fee reports **strips the `If-Match` version header** that ADR 0006/0014 require, defeating optimistic locking; the discount report rows aren't `updated_at`-touched on re-emit so `last_synced_at` is a misleading freshness signal; HMAC was removed from the HMS-to-summary-service hop (comment-acknowledged but still a defense-in-depth regression); `idx_outbox_aggregate` and the `discount_reports.invoice_no` GIN search index live only in migrations and will be wiped by `prisma db pull` per the project's stated regeneration policy; the shared fee-report table component is a 368-line god-component with 13+ props covering six concerns.

## Issues

### Critical

1. **Bulk pay/revert drops the `If-Match` version header required by the summary-service.** CLAUDE.md ADR 0006 and ADR 0014 require every CFI status change to send `If-Match: <version>` for the optimistic-lock guard, and the discount/migration reads as not yet in production yet still inherits the same row-version discipline from the `version` column that ships with every report. The new pay/revert schemas (`payCfReportSchema`, `payIhdReportSchema`, `payPfReportSchema`, `payReadingReportSchema`, `payRfReportSchema`, `payTcReportSchema`) take an `ids: string[]` array but no `versions: string[]`, and the proxy routes (`src/app/api/(common)/reports/*/pay/route.ts`, `…/revert/route.ts`) forward only `{ …body, paidById }` / `{ ids, changedById, remark }`. Net effect: the summary-service either rejects with 428 (if it enforces the header strictly) or — worse — accepts the bulk update without the lock check and silently lets two concurrent operators double-pay or revert a row that was just paid. Either way the optimistic-lock invariant from CLAUDE.md is broken end-to-end. Fix: extend every `pay*Schema`/`revert*Schema` with a parallel `versions: z.array(z.number().int().nonnegative()).min(1)` array (or a `Map<id, version>`), wire the client to send `If-Match` per row (HTTP allows only one `If-Match` so the body must carry them), and have `FeeReportTable.payFn`/`revertFn` pair every selected id with its row's `version`. Reference files: `src/app/(dashboard)/common/reports/consultation-fees-report/features/schemas/get-cf-report.schema.ts:16-31`, `src/app/api/(common)/reports/consultation-fees/pay/route.ts:18-32`, and the matching five siblings.

### High

2. **`discount_reports` migration-only `idx_discount_invoice_no_trgm` GIN index will be wiped by `prisma db pull`.** The project CLAUDE.md says the summary-service uses a hand-maintained Prisma subset regenerated by `npx prisma db pull`, and CHECK constraints / pg_trgm indexes are explicitly called out as the things that don't survive that round-trip. The new discount-reports migration adds `CREATE INDEX "idx_discount_invoice_no_trgm" … USING GIN (lower("invoice_no") gin_trgm_ops)`, but the Prisma model only declares the standard indexes. The first time someone runs `prisma db pull` to refresh the model, this index silently disappears from the canonical subset, the case-insensitive `search` filter becomes a sequential scan, and the migration history is the only place it lives. Same risk for `ck_discount_source` and `ck_discount_amounts_nonneg`. Fix: put the hand-written SQL in `hms-docs/summary-service/data-model/schema.sql` (the documented source of truth for non-Prisma-able DDL) and reference it from the migration's leading comment, so regeneration flows stay aligned. Reference files: `prisma/migrations/20260713120000_add_discount_report/migration.sql:40-46`.

3. **No `If-Match` enforcement on the proxy routes themselves.** Even after the client starts sending versions, the proxy at `src/app/api/(common)/reports/*/pay/route.ts` and `…/revert/route.ts` accepts and forwards a body without validating that `body.ids.length === body.versions.length` (or whatever shape the new schema picks). A client that sends mismatched arrays will fail at the summary-service; the proxy should fail-fast with 400. Reference files: the six `pay/route.ts` and six `revert/route.ts` files.

4. **Refund emits `serviceIds`/`procedureIds` arrays into the outbox payload, but `opd-billing-events.ts` types `extraPayload` as `Record<string, unknown>` with no shape.** The refund path is the one place the comment in `opd-billing-events.ts` says "carry data the worker can't re-derive" — meaning it's load-bearing for the consult/tele/in-house reports to know which lines were refunded. With an untyped `extraPayload`, a future maintainer can rename the fields, miss the consumer, and silently ship a refund that doesn't void the right rows. Fix: replace `Record<string, unknown>` with a discriminated union or at minimum a `RefundExtraPayload` interface, and document it as the contract. Reference files: `src/lib/opd-billing-events.ts:36-53`, `src/app/(dashboard)/shared/opd/services/refund/opd-refund.service.ts:73-89`.

5. **`source` enum coverage mismatch between migrations and schemas.** IHD, PF, RF, and reading reports use a `source` column with a CHECK constraint `'OPD' | 'IPD'` (and `'LAB' | 'IMAGING'` for reading). The shared UI types in `fee-report-shared/types.ts` only model payout status / adjustment type / mode — they don't include `source`. Meanwhile the discount report uses `source` ('OPD' | 'IPD') but has **no** `BillingDate` index on `(source, billingDate)` — only on `(tenant_id, billing_date)`. Once a tenant ships meaningful OPD + IPD volumes, the per-source filter that the API route accepts (`if (query.invoiceType) sp.set("invoiceType", query.invoiceType);`) will trigger a full scan of the date-range partition. Fix: add `@@index([tenantId, source, billingDate(sort: Desc)])` (or equivalent) for IHD/PF/RF/reading/discount, and at least model `source` in the shared types so all six reports type-check uniformly.

6. **HMAC removed from HMS-to-summary-service without a documented compensating control.** `src/lib/summary-api.ts:5-9` says: "Server-to-service auth was removed (summary-service v1). The summary-service tenant-guard middleware only requires a plain `X-Tenant-Id` header (validated as a UUID). v2 will reintroduce an auth context; until then the header is trusted on the wire (both services are on the same private network)." CLAUDE.md ADR 0008 mandates HMAC-SHA256 with `X-Service-Id`/`X-Signature`/`X-Timestamp`/replay-cache. A misconfigured DNS, sidecar, or L7 proxy that exposes port 4000 to anything other than the HMS host turns the X-Tenant-Id header into the sole authorization (and `DEFAULT_TENANT_ID` is just a UUID, so it's not even a secret). The existing defense-in-depth was deliberately removed and not replaced. At minimum: keep the previous `hms-docs/summary-service/api/hmac-auth.md` as an authoritative spec, mark this PR's removal as a regression, and re-enable HMAC before the discount-report feature goes to a tenant with non-zero trust assumptions.

### Medium

7. **`summary-api.ts` 10s timeout is hard-coded and not configurable.** A 10s ceiling is reasonable for a list read but will silently 502 the report UI on a legitimately busy month-end. Hard-coded constants age badly. Ponytail: extract a `SUMMARY_API_TIMEOUT_MS` env override; lift it when it bites.

8. **`useEffect` URL-sync in every report page re-runs on mount with `[]` deps and `eslint-disable-next-line react-hooks/exhaustive-deps`.** Each of the six new report pages (cf, tc, ihd, pf, rf, reading, opd-income, discount — 8 pages) has the same `useEffect(() => { router.replace(...) }, [])` block that suppresses the exhaustive-deps lint rule. This is a known footgun: if `effectiveQuery` changes (it can, via `useMemo` keyed on `parsed`), the URL won't follow. Fix: rewrite as `useEffect(() => { if (no url params) router.replace(...) }, [effectiveQuery])` and drop the eslint-disable. Reference: `src/app/(dashboard)/common/reports/discount-report/page.tsx:25-32` (representative).

9. **`summaryApi.post` accepts `body: unknown` and `JSON.stringify`s anything.** A future caller passing a non-serializable value (Date, Map, BigInt) gets a runtime error inside the proxy. Fix: type `body: unknown` as `body: Record<string, unknown> | unknown[]` and pre-validate with a tiny `try { JSON.stringify }` (or use the native `Response.json` helper) to surface the failure as a 500 instead of an opaque Next.js worker crash.

10. **`discount_reports.last_synced_at` defaults to `now()` and is never updated.** The schema has `lastSyncedAt DateTime @default(now())` with no `@updatedAt` and no `updatedAt` mutation in the outbox handler (which lives in the summary-service). Once the worker reconciles a row, the field's claim of "last time we synced" is false unless the worker actively writes to it. Either drop the column or have the summary-service bump it on every reconcile. Same pattern applies to `lastSyncedAt` on the other six fee-report tables.

11. **`fee-filter-modal.tsx` calls `makeFetchDoctorsQuery({ limit: 0, offset: 0, page: 1 })` on every render of the filter modal.** If the page mounts with the modal closed this is fine, but `FeeReportTable` always renders the `FeeFilterModal`, so every report page on mount fires a doctor-list query at offset 0 — and `limit: 0` likely means "all" on the doctors endpoint, which is the entire doctors table per tenant. Confirm the doctors query accepts `limit: 0` as a sentinel for "no limit"; if it does, this is intentional; if not, it's a foot-gun. Fix: pass an explicit `limit: 100` (or whatever the existing convention is) and stop at first page.

12. **Six `pay/route.ts` files each import their own `pay*Schema` from per-report paths.** With six schemas that are byte-for-byte identical (all are `.object({ ids, adjustmentType, adjustmentMode, adjustmentValue })` with the same refine), a single `fee-report-shared/schemas/pay-report.schema.ts` would eliminate 6 copies and prevent drift. Ponytail: collapse.

13. **Bulk pay/revert fails open on per-row errors.** When the summary-service returns `{ paid: [...] }` shorter than the request (i.e. some rows were rejected), the proxy returns success and `FeeReportTable` shows the success toast. The user can't tell which rows paid and which didn't. Fix: the proxy should surface partial failures (409 with which ids failed) and the toast should say so.

14. **`fee-report-table.tsx` is a 368-line god-component** with pay modal, revert modal, activity-log modal, column visibility menu, row selection, fixed-bottom action bar with `useEffect`-driven `barLeft`/`barWidth` measurement, and sticky footer — all in one file. Each concern is independently testable. Ponytail: extract `useSelectionActionBar` (the barLeft/barWidth logic is the most non-trivial piece) and split modals into separate files. At minimum, move the `useEffect`-on-resize into a `useStickyBarGeometry()` hook so it can be unit-tested.

### Low / Nit

15. **`money = (n) => n == null ? "-" : n.toLocaleString("en-US")` is duplicated** in three places (`fee-report-columns.tsx:7`, `fee-report-table.tsx:24`, `discount-report-table.tsx:14`). Ponytail: hoist to `@/utils/money.ts`.

16. **`discount-report-table.tsx` exports `DiscountReportTable` as a named export** while every other report uses `ConsultationFeesReportTable` / `TeleConsultantFeesReportTable` etc. Consistency: rename to `DiscountReportPageTable` or accept the divergence. (Nit, but the lack of consistency across 8 sibling pages is what makes a small table accidentally look like the shared component.)

17. **Hard-coded date-format strings (`"D MMM YYYY"`, `"h:mm A"`) are repeated in at least 5 files.** Hoist to `@/utils/date-format.ts` constants.

18. **`STATUS_OPTIONS`, `INVOICE_TYPE_OPTIONS`, `SOURCE_OPTIONS`** in `fee-filter-modal.tsx` are inline; they belong with the shared `fee-report-shared/types.ts` since the Zod enums on the API side reference them.

19. **`.env.example` diff: 4 added, 1 deleted** — check that the deleted line wasn't a real existing env var being removed. (`SUMMARY_API_URL` is a new dep; if the deletion removed an unrelated var it's a regression in another feature.)

## Recommendation

The headline blocker is **#1 (missing `If-Match` for bulk pay/revert)** — it's a direct violation of CLAUDE.md ADRs and silently breaks optimistic locking across all six fee-report types. **#2 (Prisma-regeneration risk for the GIN index and CHECK constraints)** is the second blocker because the project's stated workflow makes this index vanish on the next `db pull`.

Before merging:

- Wire `versions: number[]` (or a parallel array) through `pay*Schema` / `revert*Schema`, the six pay/revert proxy routes, the `FeeReportTable.payFn`/`revertFn`, and the `FeeReportRow` type. Add server-side mismatch validation (400).
- Move the discount-reports hand-written SQL (CHECK constraints + pg_trgm GIN) into `hms-docs/summary-service/data-model/schema.sql` so the next `prisma db pull` doesn't wipe it.
- Type the refund payload in `opd-billing-events.ts` — it's the one piece of data the worker can't re-derive.
- Either restore HMAC on the HMS→summary-service hop or document the compensating control as a release blocker.
- Add the missing `(tenantId, source, billingDate)` composite index for IHD/PF/RF/reading/discount.
- Decide whether `lastSyncedAt` is load-bearing — drop it or bump it on reconcile.

Nice-to-haves for the next pass: collapse the six identical pay schemas, split the 368-line shared table, and hoist the `money`/date-format utilities.

Not blocking but worth a follow-up: read `next.config.ts` `generateRouteRewrites` to confirm all six new sidebar entries route correctly — I checked that the mapping pattern works for the existing reports, but the PR doesn't update `route-mapping.ts` if the rewrite generator derives from a file list.
