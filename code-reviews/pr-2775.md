## Summary

This PR adds an appointment-scoped activity-log read endpoint and flips the patient-detail appointment card to read-only mode. The substantive work is the new file `src/app/api/(appointment)/appointments/[id]/activities/route.ts` (45 lines), a 4-line addition of `readOnly` to `<AppointmentCard>`'s props at `src/app/(dashboard)/common/patients/features/components/patient-appointments-tab.tsx:131-135`, and two submodule pointer bumps (`src/app/(dashboard)/appointment` from `499443262` to `bcfd87fcc`, `src/app/(dashboard)/opd` from `ddc0b54c` to `f6c0cef8`).

The new route is correct in shape: it uses `enhancedApiHandler({ auth: { required: true } })` (the codebase's preferred wrapper — see `src/app/api/(appointment)/appointments/route.ts:6`), queries `prisma.activityLog.findMany` filtered by `entity: "Appointment"` and `entityId: id`, ordered by `timestamp: desc`, and includes the same `user → role` projection used by `src/app/(dashboard)/common/reports/activity-logs/features/activity-logs.service.ts:54-69`. The `AppointmentDetailModal` referenced by `AppointmentCard` already exists at `src/app/(dashboard)/appointment/appointment-list/features/components/modals/appointment-detail-modal.tsx`, so the `readOnly` flag is presumably used to hide the `AppointmentStatusSelect` / `EditAppointmentModal` triggers in `appointment-card.tsx:281-296` — that change lives inside the submodule pointer bump and is not directly visible from this PR's diff.

The endpoint follows the project's recent convention for entity-scoped activity-log queries (compare `src/app/api/(pharmacy)/purchased-price-updates/[id]/activity-logs/route.ts:5-25`), but it has three concerns: (1) **no action-level permission check** — `auth: { required: true }` only proves the user has a session; any logged-in user can read the full audit history of any appointment, including those belonging to other clinics/doctors, because the existing `Appointment List` permission module at `permission-ui-config.ts:213-215` defines `view` as the only read action and there is no `permissions: [...]` array on the route; (2) **no `id` UUID validation** — `params.id` is passed straight to Prisma without a `.uuid()` Zod check, which will surface as a Postgres `::uuid` cast error for malformed paths like `/api/appointments/abc/activities`; and (3) **no pagination / cap** — `findMany` has no `take`, so a long-lived appointment (which can accumulate hundreds of edits) returns the entire history in one response. The new code is otherwise clean: no `console.log`, no `@ts-ignore`, no `eslint-disable`, no `any`, no non-null `!`, no outbox/transaction concerns (read-only endpoint), no Prisma migration.

## Issues

### High

- **`src/app/api/(appointment)/appointments/[id]/activities/route.ts:6-9` — No action-level permission check on a read endpoint that returns PII (patient name, role, username)**
  `auth: { required: true }` only proves the user has a session (`api-handler.ts:139-144` — `verifyApiAuth()` is the only call). Any logged-in user (e.g. a receptionist with only `View` on their own module) can fetch the full audit history of any appointment, which exposes `username`, `fullName`, and `role.name` of every user who ever touched the record (`route.ts:21-36`). The `Appointment List` module at `permission-ui-config.ts:213-215` defines a `view` action (it's the only action in `crudExport` minus `delete`/`isDepositRequired`), and the page-level `AppointmentCard` already gates the detail-modal action icon with `<PermissionGuard action="View" subject="Appointment List">` (`appointment-card.tsx:265, 282`). The API route must mirror this gate. Fix: add `permissions: [{ action: "View", subject: "Appointment List" }]` to the `auth` block:
  ```ts
  auth: {
    required: true,
    permissions: [{ action: "View", subject: "Appointment List" }],
  },
  ```
  Without this, the page guard is decorative — the browser-side `PermissionGuard` is bypassable, and any logged-in session can hit the route directly.

- **`src/app/api/(appointment)/appointments/[id]/activities/route.ts:11-14` — `params.id` is not UUID-validated; malformed paths surface as raw Postgres cast errors**
  `const { id } = await params;` then `entityId: id` — there is no `.uuid()` Zod check. The Prisma column `entity_id` is `@db.Uuid` (`prisma/schema.prisma:1839`), so a request like `GET /api/appointments/not-a-uuid/activities` will throw a Prisma raw error (`Invalid value for argument 'entityId'. Must be a UUID.`), which the global error handler in `api-handler.ts:37` logs and surfaces as a 500 with `"Internal Server Error"` masking the real cause. Add `validation: { params: z.object({ id: z.string().uuid() }) }` to the handler options, or inline a guard:
  ```ts
  const { id } = await params;
  if (!z.string().uuid().safeParse(id).success) {
    throw new AppError("Invalid appointment id", 400);
  }
  ```
  The neighboring `purchased-price-updates/[id]/activity-logs/route.ts:5-10` has the same hole — worth opening a follow-up to fix both.

### Medium

- **`src/app/api/(appointment)/appointments/[id]/activities/route.ts:13-41` — Unbounded `findMany` with no `take` / cursor pagination**
  `prisma.activityLog.findMany({ where: ..., orderBy: ..., include: ... })` has no `take` or `skip`. A long-lived appointment (a follow-up patient over years, or an appointment that gets rescheduled many times) can accumulate dozens-to-hundreds of edit-log rows. The response is unbounded; on a `/api/.../activities` cold-start with 500 rows this will serialize all of them with a 4-deep nested `user → role` include. The activity-log module at `src/app/(dashboard)/common/reports/activity-logs/features/schemas/get-activity-logs.ts` already has a `limit`/`offset` cursor convention (verified at `activity-logs.service.ts:48-49` — `take: query.limit || 10, skip: query.offset || 0`). Mirror it: accept `limit` / `offset` (or `take` / `cursor`) query params and pass them through `findMany`. At minimum, set a hard cap like `take: 100` to prevent the worst case until pagination lands.

- **`src/app/api/(appointment)/appointments/[id]/activities/route.ts:13-41` — No existence check on the appointment before returning the audit log**
  The route returns `{ success: true, result: activityLogs }` even when `id` does not correspond to a real appointment. This is a soft existence-leak: a caller can distinguish "appointment does not exist" (empty `result` array) from "appointment exists but has no edits" (also empty `result` array). More importantly, an attacker can iterate UUIDs to discover which IDs are real appointments by side-channel timing differences (the `findMany` returns fast on a non-existent FK vs. slow on a real one with many rows). The repo's API-handler convention is to return 404 when the parent resource is missing (see `purchased-price-updates/[id]/activity-logs/route.ts:13-16`, which calls a service that does an `appointmentService.findById` first). Add a `prisma.appointment.findUnique({ where: { id }, select: { id: true } })` and `throw new AppError("Not found", 404)` if missing — the consistent 404 response also stops the existence-leak.

- **`src/app/api/(appointment)/appointments/[id]/activities/route.ts:21-36` — N+1 / join pattern is fine here, but the `role.name` projection is duplicated 3x in the codebase**
  The `include: { user: { select: { id, username, fullName, userId, role: { select: { id, name } } } } }` shape is repeated verbatim at `src/app/(dashboard)/common/reports/activity-logs/features/activity-logs.service.ts:54-69` and (almost) at `src/app/api/(pharmacy)/purchased-price-updates/[id]/activity-logs/route.ts:13-16` (which delegates to a service that does the same). Not a bug — just a candidate for a shared `ACTIVITY_LOG_USER_INCLUDE` constant in `src/utils/prisma-includes.ts` (or wherever the codebase keeps shared `include` shapes). Follow-up.
