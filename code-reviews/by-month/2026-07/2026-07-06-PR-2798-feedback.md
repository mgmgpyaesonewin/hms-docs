# PR #2798 Review: Inhouse doctor fees report for ipd and opd

**Repo:** MyanCare/Ycare-HMS
**PR URL:** https://github.com/MyanCare/Ycare-HMS/pull/2798
**Author:** myopaingthu
**State:** OPEN
**Verdict:** changes-requested

## Summary

Adds an In-house Doctor Fees report that unifies OPD + IPD line items, a Consultation Fees report, and a Tele Consultant Fees report, all backed by a new `event_outbox` table, two shared `enqueue*Event` helpers, and a thin Next.js API layer that proxies to the existing `summary-api`. The schema, repository emits, and outbox plumbing are sound, but the PR is bloated by ~3x copy-paste of the three report surfaces (table, modals, schemas, types, API routes), ships a few real correctness bugs (broken `search` filter, missing transaction around activity-log writes, env-driven tenant_id), and the 5-migration change should be a single migration. The design intent is good — the IHD unification across 4 IPD line tables is genuinely well-thought-out — but the implementation needs a significant dedup pass and a couple of fixes before merge.

## Findings

### Blocking

1. **Search filter is broken end-to-end.** `get-cf-report.schema.ts:18`, `get-tc-report.schema.ts:18`, `get-ihd-report.schema.ts:19` accept `search` and the client sends it (`cf-report.api.ts:34-37`), but the GET route never forwards it to summary-api (`src/app/api/(common)/reports/consultation-fees/route.ts:18-23` only forwards `start`/`end`/`status`/`doctorId`). Either remove `search` from the client + schema (and the DataTableSearchbox) or actually pass `search` through to `/cf-fee-reports?...`. Same omission in the TC and IHD route files.

2. **Outbox tenant_id is read from env, not from the bill.** `src/lib/opd-billing-events.ts:24-30` and `src/lib/ipd-daily-bill-events.ts:24-30` both do `process.env.DEFAULT_TENANT_ID?.trim()`. This silently drops the event if the env var is unset, and worse, writes the wrong tenant_id when one is set (multi-tenant HMS, single-tenant summary service). Either (a) resolve the bill's `tenantId` inside the transaction and pass it explicitly into `enqueue*Event(tx, eventType, billId, tenantId, ...)`, or (b) drop the outbox envelope's `tenantId` column and let the worker look it up from `aggregateId`. Option (a) is the only one that matches the multi-tenant claim in `hms-docs/`.

3. **Activity-log writes are not in any transaction with the pay/revert.** `src/app/api/(common)/reports/consultation-fees/pay/route.ts:30-41` and the IHD/TC counterparts do `summaryApi.post(...)` and then `prisma.activityLog.createMany(...)` in two separate steps. A summary-api success + BFF crash + retry = double-counted activity log entries. Either fold the activity-log write into a single `$transaction` (and accept that summary-api is a remote call best-effort, so the existing try/catch is fine — but then make the failure explicit in `activityLogFailed`, which it already does) OR call summary-api first inside a $transaction that includes the activity-log createMany. Today the contract "pay succeeded, log may or may not exist" is documented only in the toast; the API returns `activityLogFailed: true` to the client but the client renders that as a yellow warning, not an error — operators won't notice.

### Important

1. **5 migrations for one logical change.** The PR adds `20260617091918_add_summary_service_tables`, `20260618090000_add_tc_fee_report_tables`, `20260623120000_add_ihd_fee_report_tables`, `20260623130000_add_ihd_ward_service_ref`, `20260624120000_add_ihd_proxy_cathlab_refs`. Only the first and last two are different in any way that can't be merged. The CF, TC, and IHD table trios are structurally identical (3 tables each: `*_fee_reports`, `*_fee_report_status_changes`, `*_fee_report_adjustments` with the same CHECK constraints and indexes). Squash into one `add_summary_service_reports` migration that creates `event_outbox` + all 3 report trios + the IHD ward-service/proxy/cathlab grain columns. Migrations are append-only, so rebase and merge.

2. **Three near-identical report surfaces = ~1,800 lines of copy-paste.** `consultation-fees-report-table.tsx`, `in-house-doctor-fees-report-table.tsx`, `tele-consultant-fees-report-table.tsx` are 369 / 370 / 369 lines, byte-for-byte identical except for the row type, query-key string, money/format/export deltas, and the two schemas that have a `source`/`invoiceType` field. Same for `cf-pay-modal.tsx` vs `ihd-pay-modal.tsx` vs `tc-pay-modal.tsx` (112 lines each, identical), the three `*-revert-confirm.tsx` (72 lines each, identical), the three `*-activity-log-modal.tsx` (80 lines each, identical), the three `*-filter-modal.tsx` (69/87/69 — only the IHD variant adds the `invoiceType` select), the three `*-report-columns.tsx` (164 lines each, identical except for the data accessor and the "Consultation Fees"/"Tele Consultant Fees"/"In-house Doctor Fees" header), and the three `*-report.types.ts` (59/66/59). Extract a `components/reports/fee-report/` package with one `FeeReportTable<T>` component parameterized by a config object (`{ kind, rowType, feeAccessor, feeHeader, allowInvoiceType, queryKey, schemas, base }`); pull the three columns/modal/filter into shared modules that take a small bag of overrides. This will reduce ~2,800 lines of new code to ~900. The right time to do this is before merge, not "later" — the third copy is where the duplication pattern locks in.

3. **Empty `NumberInput` value silently becomes 0 on submit.** `cf-pay-modal.tsx:32-33` stores `percent` and `amount` as `number | string`, then `handleSave` does `Number(percent)`. If the user clears the field (which is allowed by the empty-string `min={0}` configuration), the value is `""` and `Number("")` is `0`, so a partially-filled form submits `adjustmentValue: 0` and silently applies a 0% / 0-amount adjustment. Either coerce via `Number(percent) || undefined` and refuse to submit when both are empty, or maintain a separate `hasPercent` / `hasAmount` flag set on first non-empty input.

4. **`cancelOPDBilling` and `deleteOPDBilling` now wrap a single op in `prisma.$transaction`.** `opd-billing.repository.ts:635-652` and `:1838-1851` introduce `$transaction` for what is a single-row write + outbox emit. The previous code was a single `prisma.oPDBillingService.update` with no transaction. Wrapping in `$transaction` is fine (and arguably correct for the outbox), but the PR is the only place doing this and the surrounding call sites are not updated — confirm there is no caller that was relying on the no-transaction semantics (e.g. running inside its own outer transaction). If any caller already has a tx, nesting `prisma.$transaction` will deadlock on the connection pool.

5. **`createProxyBillLines` and `createProxyBillWithLogic` emit the same outbox event.** `proxy-bill-template.service.ts:251-265` emits `ipd_daily_bill.bill_reconciled` in `createServiceDailyBill`, then `:303-318` emits it again in `createProxyBillWithLogic` for the same IPD proxy bill. The result is two outbox rows per IPD proxy-bill creation → the summary-service worker re-derives twice and writes a second outbox update on the second pass. Pick one emit site (the one with the confirmed `ipdDailyBillId`) and delete the other.

6. **`/api/reports/{cf,tc,ihd}-fees` returns ALL rows for the date range with no pagination.** `cf-report.api.ts:21-24` and the route handler at `src/app/api/(common)/reports/consultation-fees/route.ts:23-29` ignore `limit`/`offset` and the summary-api `/cf-fee-reports` returns the entire unfiltered-for-the-date-range set. For a hospital billing a year's worth of consultations per doctor, this is O(millions of rows) over a remote service call with no streaming. The schema already has cursor pagination in the CFI worker (per the hms-docs), so the boundary is just choosing to use it. Cap the page size server-side at the very least.

7. **Manual `containerRef` + `useState(barLeft/barWidth)` + `resize` listener for the action-bar position** is reinventing `position: sticky; bottom: 0` inside an existing scroll container. `consultation-fees-report-table.tsx:90-104` and the IHD/TC copies maintain a fixed-position bar synced to the parent rect via state, re-measuring on every resize. CSS `position: sticky` (already used for the table header at line 166) handles this in the browser with no JS, no resize listener, no measurement ref, no React state. Replace 15 lines of imperative code with `<Box style={{ position: "sticky", bottom: 0, ... }}>`.

8. **Date-range default race.** `page.tsx:38-46` sets the URL to today's range inside a `useEffect` and renders `useSuspenseQuery(makeCfReportQuery(effectiveQuery))` from the same data in the same render. The first render uses the effectiveQuery (today), the useEffect rewrites the URL to encode the same range, and the URL is then the source of truth. This works in practice but it's three sources of truth (URL, state, default) where one would do. Either default at the server side (preferred — the page is `("use client")` for no good reason; the table can be the only client component), or compute `effectiveQuery` purely from the URL inside `useMemo` and use `router.replace` only when the URL is missing both keys (today's branch), which is what's already there. The `// eslint-disable-next-line react-hooks/exhaustive-deps` is the smell.

9. **Permission check at `View` for `POST /pay` and `POST /revert`.** `src/app/api/(common)/reports/consultation-fees/pay/route.ts:21` requires only `View` to bulk-mutate payouts. This is probably a typo for `Pay` / `Revert` (or whatever action the permission system maps to "execute payout"). Verify against the permission module config added in `permission-ui-config.ts:892-906`, which only defines the `View`-only sub-modules — the `Pay` / `Revert` actions are not declared anywhere in the diff. Either add them to the permission config, or use an existing one.

### Nit

1. `cf-report.api.ts:3-10` builds the querystring by hand when `URLSearchParams` (already imported for `sp`) could be used the same way the API route handler does it (`src/app/api/(common)/reports/consultation-fees/route.ts:15-20`). The two implementations should not drift.

2. `cf-report.types.ts:32-37` and the IHD/TC twins re-declare `CfAdjustmentType`/`CfAdjustmentMode`/`CfPayoutStatus` per type file. The enum is the same `("UNPAID" | "PAID" | "PLUS" | "MINUS" | "FULL" | "PERCENT" | "AMOUNT")` in all three files. Promote to `types/fee-report.enums.ts`.

3. `revertCount` is incremented server-side (implied — there is no code path in the diff that does it) and rendered as the "Unpaid (Reverted)" label. The `cf-report-columns.tsx:73-77` `StatusBadge` does the right thing but couples the read to a counter the client cannot audit. Render `sourceVoided` only, or expose a `lastRevertedAt` and let the label follow the date.

4. `migrations/20260617091918_add_summary_service_tables/migration.sql:39` `consultation_fee INTEGER` and the matching `in_house_doctor_fee` / `fee` columns are `INTEGER` (not `NUMERIC`). The summary-service docs (`hms-docs/summary-service/README.md`) explicitly say money is stored as `NUMERIC(12,2)`. This is a real money/precision issue, not a code review nit — demote this to "Important" if confirmed against the docs; otherwise it's a "needs confirmation".

5. The CF migration's `pg_trgm` GIN index uses `lower("invoice_no")` while the existing summary-service worker (per ADR 0009) uses `lower(invoice_no) % arbitrary_string`. Match the existing pattern; the new path will be searched by the same code.

6. `opd-billing-events.ts:32-35` logs a warning and silently returns when `DEFAULT_TENANT_ID` is missing. In production this is exactly the bug class the outbox is supposed to prevent (silently dropped events). Throw or assert, don't warn.

7. `consultation-fees-report-table.tsx:55-63` `payMut.onSuccess` / `revertMut.onSuccess` both toast on `activityLogFailed: true`. The toast message says "Payment recorded, but the activity log entry failed to save" — this is silently degraded behavior shown to a user who has no way to do anything about it. Either log the failure to a channel operators can see, or surface it as a real error with a retry button.

8. `permission-ui-config.ts:892-906` adds three sub-modules to "Doctor Report" with `excludeActions: ["add", "edit", "delete"]`. The other reports in the file (Activity Report, etc.) follow the same pattern — fine. The "Doctor" sidebar group label is generic enough to be confused with the existing "Doctor" group; verify the sidebar (`sidebar-link-config.ts:798-813`) is the only place "Doctor" appears.

9. `.env.example:5` adds `SUMMARY_API_URL=http://summary-api:4000` which is the Docker-internal hostname. The README/HMS-app local dev typically uses `http://localhost:4000` — confirm the README still works for the case where the user runs the summary-service natively on the host.

10. `summary-api.ts:9-13` calls `getBaseUrl()` on every request. The URL doesn't change at runtime; cache it as a module-level constant.

11. The `additionalData: { remark }` on revert (`cf pay/revert/route.ts:42`) and the IHD/TC counterparts write JSON to the activity log. The activity log modal reads `additionalData.remark`. The data shape is implicit; the schema is `prisma.activityLog.additionalData Json` per the existing model. No code change needed, but add a type assertion in the API handler so the contract is explicit.

12. The 3 IHD migration columns (`ipd_ward_service_item_id`, `ipd_service_bill_item_id`, `ipd_cath_lab_service_item_id`) are added in 2 separate migrations. They're all "nullable unique refs to per-line IPD tables that can host an in-house-doctor service". One migration with all three columns and three unique indexes would have been sufficient.

### Question

1. The `event_outbox` migration adds a partial index `idx_outbox_pending` on `(next_attempt_at) WHERE status = 'PENDING'`. The schema.prisma declares it as `@@index([nextAttemptAt], map: "idx_outbox_pending")` without the `WHERE` clause. Prisma will run `prisma migrate dev` against the canonical schema and **drop** the partial index because Prisma's introspection of an unmanaged index doesn't preserve the predicate. Either drop the Prisma-side `@@index` (keep the migration-only partial index and document the divergence) or add a Prisma `@@index` with `where: { status: PENDING }` if Prisma supports it for the version in use. Confirm before merge.

2. `in-house-doctor-fees` re-uses the same `idx_cf_tenant_billing_date`-style index on `tenant_id` + `billing_date`. Is the IHD list page going to query by date range + tenant most of the time? If yes, the index is correct. If the typical IHD query is "everything for one doctor across all time" (i.e. `WHERE doctor_id = ?`), the current `(tenant_id, billing_date DESC)` composite is wrong (Postgres will scan the whole index). Consider a `(tenant_id, doctor_id, billing_date DESC)` index for IHD specifically.

3. The summary-api worker (separate repo, not in this diff) is the consumer of `event_outbox`. The two repos need to stay in sync on event types (`opd_billing.bill_created`, `opd_billing.bill_reconciled`, `opd_billing.refund_created`, `ipd_daily_bill.bill_created`, `ipd_daily_bill.bill_reconciled`). Where is the source of truth for this enum, and is there a CI check that fails when one side adds an event the other doesn't handle? This PR adds 5 event-type strings across 2 files with no companion test.

4. The PR includes the CF and TC reports that are not mentioned in the title or the ClickUp ticket (https://app.clickup.com/t/86exqc30g). Confirm scope: is the intent to ship all 3 in this PR, or did CF/TC sneak in as scope creep?

5. `revertCount` is read by the client (`cf-report-columns.tsx:75`) to label "Unpaid (Reverted)". The summary-api increments it on revert. Is there a documented contract that this counter is monotonically increasing, or is the client reading a derived field the worker may re-derive (and reset) on a full reconciliation pass? The summary-service docs say "the worker re-derives from the bill" — what happens to `revert_count` when a bill is reconciled?

## Recommendation

The IHD unification across the 4 IPD line tables is genuinely well-modeled (CHECK constraints, partial unique indexes per channel, a single `source` enum). Hold the PR for: (1) fix the broken `search` filter, (2) decide the tenant_id resolution for the outbox (the env-var approach is a tenancy bug), (3) collapse the 3x copy-paste of the report surfaces into a shared `fee-report/` package, (4) squash the 5 migrations, (5) remove the duplicate outbox emit in proxy-bill. The `NumberInput → 0 on empty` bug and the `cancelOPDBilling` transaction change should also be confirmed. After those changes, this is a strong foundation — the design (transactional outbox, summary-service split, server-computed grand total) is the right shape. The "ship all 3 reports in one PR" choice is defensible only if the dedup happens in this PR; if not, land IHD alone and queue CF/TC as follow-ups that share the new `fee-report/` package.
