# PR #2830 Review: Procedure fees report

**Repo:** MyanCare/Ycare-HMS
**PR URL:** https://github.com/MyanCare/Ycare-HMS/pull/2830
**Author:** myopaingthu (Myo Paing Thu)
**Branch:** feat/procedure-fees-report -> development
**Stats:** +6870 / -21 across 84 files
**Verdict:** changes-requested

## Summary

PR adds four "fees report" UIs and APIs (consultation, tele-consultant, in-house doctor, procedure) backed by four new Prisma models with status-change and adjustment join tables, plus an `event_outbox` table and three helper services to emit the outbox events. The shape of the four reports is ~80% copy-paste; the data table is hand-rolled when the codebase already ships shared `@/components/data-table` primitives; five migrations do the work of one; CHECK/TS/Zod enums are repeated four times. Functional design is sound (materialized, event-driven, with optimistic locking and outbox) but the implementation is heavy enough that the diff can lose 2,500-3,000 lines without losing capability. Recommend landing in two stages: (1) factor the four reports into a single parameterized component/hook; (2) squash the 5 migrations to 1, then re-review.

## Findings

### Blocking

1. **Four near-identical report modules — factor before merge.** The CF, TC, IHD, and PF report folders each ship the same eight files (api, activity-log-modal, filter-modal, pay-modal, report-columns, revert-confirm, report-table, schemas, types) with the only meaningful variation being field names and labels. `cf-report-table.tsx:1731-2041`, `in-house-doctor-fees-report-table.tsx:2960+`, and the equivalent TC and PF tables are 370 lines each with `useState`, `useMutation`, `useReactTable`, the sticky `containerRef` action bar, the grand-total footer, the `money()`/`formatDate()`/`adjustmentExport()` helpers, and the `payMut`/`revertMut` lifecycle duplicated verbatim. Extract one `useFeeReportTable<TKey, TRow>(opts)` hook and one `<FeeReportTable>` component; the four reports become a few-line configuration object each. **Fix:** lift `api`, `columns`, `filter-modal`, `pay-modal`, `revert-confirm`, `activity-log-modal`, `schemas`, `types`, `table` to a single `features/` folder with a per-report config (`{ rowType, label, apiBase, amountField, source: 'OPD' | 'IPD' | 'BOTH', statusEnum }`); render via `<FeeReportTable config={cfConfig} />`.

2. **Reinvented data table — use the shared components.** `cf-report-table.tsx:1700-2041` (and 3 copies) reimplement a header, sticky-footer grand total, column-visibility menu, and bottom action bar using `useReactTable` + raw `<Table>` JSX + `useRef`/`getBoundingClientRect` to position a fixed bar. The codebase already ships `DataTableHeader`, `DataTableActionsGroup`, `DataTableExportAction`, `DataTableSearchbox`, `DataTableDatePicker`, `ClientDataTable` from `@/components/data-table` (imported in the new code at lines 1845-1896 — and *only* the header parts are used; the table body, footer, and action bar are rolled by hand). **Fix:** delete the custom `<Table>...</Table>` body, sticky `Table.Tfoot`, the `containerRef`/`useEffect(barLeft, barWidth)` effect, and the bottom `<Box style={{ position: "fixed" ... }}>` action bar. Render the body via the shared `ClientDataTable`; use Mantine `<Affix>` for the action bar; pass the grand total through a shared footer slot.

3. **`useEffect` + `router.replace` + `eslint-disable` for date defaults.** `consultation-fees-report/page.tsx:2189-2197` (and 3 copies) — the page reads `searchParams`, then fires `useEffect` once on mount with `eslint-disable-next-line react-hooks/exhaustive-deps` to push today's `start`/`end` into the URL. The `useSuspenseQuery` on line 2199 fires before the URL is updated, so the first request omits the range; the URL update then re-fires. The "default to today" should be inside the schema (`start: z.coerce.date().default(() => dayjs().startOf("day").toDate())`) or inside `validateSearchParams`. The effect-and-`router.replace` pattern is a race; it can be deleted. **Fix:** add `.default()` to the Zod schema and drop the effect.

### Important

4. **5 migrations where 1 would do.** `20260617091918_add_summary_service_tables`, `20260618090000_add_tc_fee_report_tables`, `20260623120000_add_ihd_fee_report_tables`, `20260623130000_add_ihd_ward_service_ref` (lines 451-468, 12 lines, single column + index), `20260624120000_add_ihd_proxy_cathlab_refs` (lines 469-488, 14 lines, two columns + two indexes), `20260626050919_add_pf_fee_report_tables`. The 3 ALTER-only migrations were almost certainly created by `prisma migrate dev` when an extra column was discovered mid-draft. Squash to a single `20260617000000_add_procedure_fees_reports` migration that contains `event_outbox` + all four report tables + all `*_status_changes` and `*_adjustments` join tables + all the `*_id` nullable unique indexes for IHD/PF channels. Local-dev is unaffected, and the migration log reads as "one feature" not "one feature plus three follow-ups".

5. **Enum value sets repeated 4× — single source of truth, or use Prisma enums.** The sets `('UNPAID','PAID')`, `('PLUS','MINUS','FULL')`, `('PERCENT','AMOUNT')`, `('OPD','IPD')`, `('CANCELLED','REFUNDED','REMOVED_ON_EDIT')` appear in: (a) the Prisma schema (as `String` — line 693, 772, 866, 957), (b) the hand-added CHECK constraints in every migration (e.g. `cf_fee_reports/migration.sql:159`), (c) the front-end TS unions (`cf-report.types.ts:2093-2095`, `ihd-report.types.ts` — verified pattern), (d) the Zod schemas (`get-cf-report.schema.ts:2057,2067-2068`, `get-ihd-report.schema.ts` — verified pattern). Pick one of: (i) define `enum CfAdjustmentType { PLUS MINUS FULL }` in Prisma — drops the CHECK, drops the TS union, drops the Zod enum, and Prisma generates the SQL; or (ii) extract a single `const ADJUSTMENT_TYPES = ['PLUS','MINUS','FULL'] as const` shared by TS and Zod and emit the CHECK from a generator. The current design guarantees the four will drift the first time someone adds `HALF`.

6. **`activityLogFailed` partial-success flag is a correctness smell, not a feature.** Front-end `cf-report.api.ts:1077-1096` (and 3 copies) returns `{ paid, activityLogFailed: true }` with a yellow "Payment recorded, but the activity log entry failed to save" toast at `cf-report-table.tsx:1788-1794`. The audit-trail failure is **not** recoverable: a pay-out is committed, the user is told the row is paid, but the audit log has no entry. The summary-service does this properly (HMAC, retry, etc.); the API route that mutates the materialized report should put the status change *and* the activity log write in one Prisma transaction. If the activity log is best-effort, document that. **Fix:** wrap the `cfFeeReport.update` + `activityLog.create` (or the equivalent direct insert) in `prisma.$transaction`; drop the `activityLogFailed` flag and the yellow-toast branch.

7. **CHECK constraints and indexes duplicated in Prisma and raw SQL.** Every CHECK in `cf_fee_reports/migration.sql:152-177` (and the 3 sibling migrations) is hand-written; the index `idx_cf_tenant_billing_date` is hand-written; the `event_outbox` partial index `idx_outbox_pending` is hand-written. The Prisma schema has no enum and no expression index, so most of this *can't* be expressed in Prisma — but the comment at `cf_fee_reports/migration.sql:103-105` ("Prisma schema cannot express a filtered index") is the only place that says so. Either: (a) use `prisma migrate diff --from-empty` from a Prisma schema that *can* express what you need (Postgres enums + `@@index(..., type: Brin)` — partial indexes are still limited), then hand-patch the result; or (b) put the hand-added statements in a single trailing block per migration with a clear `-- Hand-added` separator (already done) **and** a `docs/adr/` entry that says which parts of the schema must be hand-patched. As written, the next person to add a column will re-derive the CHECK set by hand and miss one.

8. **No visible tenant-scope enforcement on the front-end.** `.env.example:13-14` adds `SUMMARY_API_URL` and `DEFAULT_TENANT_ID`. The front-end `cf-report.api.ts:1037` (and 3 copies) hardcodes `fetch("/api/reports/consultation-fees", { method: "GET" })` — no `X-Tenant-Id` header. Either: (a) confirm the front-end never needs `X-Tenant-Id` (single-tenant), in which case the env var is unused and can be removed from the front-end scope; or (b) thread the tenant from `useTenant()`/`PermissionGuard` and pass it as a header. The summary-service is multi-tenant by design (CLAUDE.md: "Multi-tenancy is defense-in-depth"); if the front-end is multi-tenant, this is a missing header.

9. **`useSuspenseQuery` will throw to the nearest error boundary on every 4xx/5xx.** `cf-report-page.tsx:2199`, and the IHD/TC/PF equivalents, use `useSuspenseQuery`. Combined with the `<button onClick={...}>` flow that does not surface `ApiResponse` errors well, the user gets a "Something went wrong" toast at `cf-report-table.tsx:1781` (which calls `toast.error({ message: err.error || "Something went wrong" })` — but `err.error` is `unknown` from `err instanceof ApiResponse` narrowing, and the fallback swallows the real error message in many cases). Switch to `useQuery` and render an inline error state, or wrap the suspense boundary and surface the error message verbatim.

10. **Front-end hand-rolls `useState`+`useEffect` for a URL filter — use the existing `useFilterParams` hook correctly.** `cf-filter-modal.tsx:1246-1258` (and 3 copies) uses `useFilterParams<{ doctorId; status; ... }>({ doctorId: null, status: null })`. The pattern is fine, but the same `useFilterParams<{ doctorId, status, invoiceType? }>({...})` shape is forked per report. After the refactor in finding 1, this becomes a single `<FeeReportFilterModal config={...} />`.

### Nit

11. **Dead import in `cf-report-columns.tsx:1425`** — `Checkbox` is imported but unused inside the column definitions. (Used in `consultation-fees-report-table.tsx`; the columns file re-exports nothing). Remove.

12. **`return <>-</>` as an empty cell** — `cf-report-columns.tsx:1439`, `ihd-report-columns.tsx`, and 2 copies. JSX with bare text inside a fragment is unusual; prefer `null`, `""`, or `<span>-</span>`. The `<></>` here is also likely to cause React key warnings when the cell is inside a `<Table.Td>`.

13. **`money()`, `formatDate()`, `adjustmentExport()` redefined 4× each.** `cf-report-columns.tsx:1431-1435` and `cf-report-table.tsx:1719-1729` (and 2 IHD/TC/PF copies) — same one-liners. Move to `@/utils/format.ts` next to `formatToDefaultDate` (which is already imported).

14. **`sourceVoided`, `sourceVoidedReason`, `revertCount`, `sourceVoidedAt`, `lastSyncedAt`** are exposed in the API response and the row types but never displayed beyond a tooltip on a single `⚠` glyph. Either surface them in the UI ("Voided on 12 Jun, REFUNDED") or drop them from the API response and the row type.

15. **`<Box style={{ position: "fixed", ... }}>` action bar at `cf-report-table.tsx:1972-2019`** — Mantine ships `<Affix>` for this. The hand-rolled `useRef`+`getBoundingClientRect`+`useEffect(resize)` dance at lines 1827-1841 to keep the bar aligned to the table is also unnecessary — `<Affix>` positions relative to the viewport.

16. **`adjustmentValue` of `0` is valid but indistinguishable from "user did not enter a value"** in the front-end state (`number | string = ""`). `Number(0) === 0` and `Number("")` throws; the modal logic at `cf-pay-modal.tsx:1354-1366` only fires the percent/amount branch if the string is non-empty, so a literal `0` is unreachable. A user cannot "Minus 0%". Consider explicit `null` for unset.

17. **Disable-manual-entry helpers (`disablePercent`/`disableAmount`)** at `cf-pay-modal.tsx:1348-1349` and 3 copies — the `useState("")` + `disabled` pattern could be a single `<ModeToggle mode={...} onChange={...} />` with internal logic. Minor.

18. **Prisma `@@map` on `EventOutbox` uses `event_outbox` (snake_case)** but the model field names are camelCase — verified consistent with the rest of the schema. No action.

19. **`.env.example:14` — `DEFAULT_TENANT_ID=00000000-0000-0000-0000-000000000001`** is the nil-UUID in the all-zeros form (zeros, not a "well-known sentinel"). The actual nil-UUID is `00000000-0000-0000-0000-000000000000`. The value used here (`...0001`) is fine as a default but the comment/variable name suggests a sentinel that doesn't match. Add a comment or rename.

20. **`updatedAt @updatedAt` vs. `lastSyncedAt` are both set on every Prisma update**, but the front-end never reads `lastSyncedAt` and the API does not return it. Either expose it (e.g. "last synced 5m ago") or drop the column.

### Question

21. **Why a new `event_outbox` table and not the existing `pg-boss` job system?** The HMS already uses `pg-boss` for background jobs (CLAUDE.md: "Background jobs via pg-boss"). The summary-service architecture deliberately chose a hand-rolled `SELECT ... FOR UPDATE SKIP LOCKED` outbox over BullMQ/Redis Streams/pg-boss, and the docs are explicit about it. But the HMS-side helper (`opd-billing-events.ts`, `ipd-daily-bill-events.ts`) writes to *this* `event_outbox`, not to `pg-boss`. Is there a deliberate split — `pg-boss` for HMS-internal jobs, `event_outbox` for cross-service events? If yes, document it; if no, route the events through `pg-boss` and drop the new table.

22. **Why materialize the report at all?** The 4 new tables (×3 if you count status-changes and adjustments) are denormalized snapshots of `opd_billing_services` / `ipd_daily_bill_services` / `ipd_ward_service_service_items` / `service_bill_items` / `cathlab_service_items` / `ipd_ward_procedure_items` / `ipd_procedure_bill_items` + 7 sibling tables. A read-time join with `GROUP BY doctor_id, payout_status` would be slower but accurate-by-construction. Was a benchmark done that justified the 4-table × 3-table materialization over a `VIEW` or a runtime join? If the answer is "doctors want instant filter-by-status", fine, but the docs should state that.

23. **`source: 'OPD' | 'IPD'` is a single text column on IHD and PF** but a `CHECK (source IN ('OPD','IPD'))`. Why a string and not a Prisma enum here when everywhere else the codebase uses enums? (IHD migration line 426-428, schema line 839.)

24. **Who reads `event_outbox.payload`?** The summary-service `OutboxPoller` presumably does. The schema stores `payload Json` and the helpers (`opd-billing-events.ts`, `ipd-daily-bill-events.ts`) presumably write a denormalized object. Is there a contract doc / TS type for the payload? If not, refactors to the OPD billing or IPD daily bill will silently break the worker.

25. **Why 7 nullable-unique indexes on IHD + PF (`opd_billing_*_id`, `ipd_*_id`)?** Lines 119-127 of the IHD migration, lines 561-568 of PF. The `nullable unique` trick (Postgres treats NULLs as distinct) works, but a `CHECK (source = 'OPD') AND (opd_billing_service_id IS NOT NULL)` + a `CHECK (source = 'IPD') AND (ipd_daily_bill_id IS NOT NULL)` would prevent the logical case "source=IPD, opd_billing_service_id set" without an index. The unique indexes are correct for idempotency, but the choice of *seven* indexes vs. one composite is worth a sentence in the design doc.

## Recommendation

Do not merge as-is. The PR works, but the four-report duplication (findings 1, 2, 3) and the migration sprawl (finding 4) make a 6,870-line diff that is ~50% of its size after a focused refactor. Concrete pre-merge asks: (1) factor the four `*-report-table.tsx` and the eight shared features into one parameterized component; (2) replace the hand-rolled `<Table>` body, sticky footer, and fixed action bar with `@/components/data-table` + `<Affix>`; (3) squash the 5 migrations to 1; (4) move `activityLogFailed` out of the API contract by wrapping the status change + activity log in a single transaction; (5) move the front-end date-range default into the Zod schema and drop the `useEffect`+`router.replace`; (6) pick one source of truth for the `UNPAID|PAID` / `PLUS|MINUS|FULL` / `PERCENT|AMOUNT` / `OPD|IPD` enums (Prisma enums recommended). The 5 service files (`cathlab.service.ts`, `ct-add-on-billing.service.ts`, `ipd-daily-bill.service.ts`, `service-request.service.ts`, `ward-service.service.ts`, `opd-refund.service.ts`, `proxy-bill-template.service.ts`) and the two outbox-emission helpers look correct but I could not review them in detail from the diff excerpt — please paste or link their full diff for a second pass.
