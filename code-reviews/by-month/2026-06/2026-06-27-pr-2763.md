## Summary

Trims leading/trailing whitespace from the `name` field (and `testScriptName` in test-script) on the create + update paths of six lookup-table services: `LabCollectionMethod`, `LabSample`, `LabStatus`, `LabUnit`, `TestScript`, and `ServicesService` (OPD). On every path it also rejects the empty-after-trim case with a 400. The duplicate-by-name check now uses the trimmed value, so `"CBC "` and `"CBC"` are detected as the same row.

The fix is small, targeted, and correct for the stated goal. It is consistent across all 6 services.

## H1. Zod schemas accept whitespace-only names, so the service-layer guard is load-bearing — push the trim into the schema instead

**Where:**
- `schemas/lab-sample/create-lab-sample.schema.ts:4` — `z.string().min(1)` accepts `"   "`
- `schemas/lab-collection-method/`, `schemas/lab-status/`, `schemas/lab-unit/`, `schemas/test-script/`, `schemas/service/` (the matching `Create*Schema` / `Update*Schema` for each of the 6 services)

The Zod schemas currently use `.min(1, ...)` (length-based). A payload of `"   "` passes schema validation (length 3, non-zero), reaches the service, gets trimmed to `""`, and is then rejected with a generic 400 from inside the service. Two consequences:

1. The schema-level error ("Lab Sample Name is required") fires for truly-empty strings but not for whitespace-only strings. The two error shapes are different — one comes from Zod, one from `AppError` — and the UI probably renders them through different branches. Users get an inconsistent error for "I typed three spaces".
2. The trim happens **after** validation. The client-side form sees the value it submitted (with whitespace) and never knows the server trimmed it. That is fine for correctness (we want to store the trimmed value), but it means the service-layer guard is the **only** line of defence against `"   "`. If a future caller (admin script, tRPC direct call, importer) skips the service, the bad value reaches the DB.

**Fix:** trim at the schema boundary so validation and storage agree on what the name is. This removes the duplicated `if (!trimmedName)` blocks from all six services:

```ts
// create-lab-sample.schema.ts
export const createLabSampleSchema = z.object({
  name: z.string().trim().min(1, { message: "Lab Sample Name is required" }),
});
```

Apply the same `.trim()` to the matching `update-*.schema.ts` and the OPD `service` schema. After this change the service-layer code collapses to just `name: payload.name` (the value is already trimmed by Zod), and the explicit empty-check becomes unreachable / can be deleted.

If the team prefers to keep the trim in the service (to centralise it), that is fine — but then the schemas should use a transform or a regex (`/^\s*$/`) so the error path stays in one place.

## H2. Trimming happens only at the write layer — list-view search (`contains`) is unaffected, so a row inserted with `"CBC "` pre-PR still surfaces when users search for `"cbc"`

**Where:**
- `repositories/lab-sample.repository.ts:97-102` — `contains: query.search, mode: "insensitive"`
- `repositories/lab-collection-method.repository.ts:103-110`
- `repositories/lab-status.repository.ts:18-23`
- `repositories/test-script.repository.ts:18-23`
- (the lab-unit and service repositories have equivalent `contains` filters)

These `contains` filters do **not** trim `query.search` before the ILIKE. Combined with the new service-layer behaviour this produces two real-world behaviours:

1. Going forward, newly-inserted rows always have trimmed names. The `contains` filter still works against them. Fine.
2. Rows inserted **before** this PR that have leading/trailing whitespace (the very rows this PR exists to clean up) will still match a search query that has its own whitespace, but won't match a query without whitespace. The list view will display them with the padding visible — which is exactly the bug the user is trying to get rid of.

This is the same data-backfill concern flagged in the prior review (H2 there). It is worth restating because the fix is small and lives entirely in the data layer:

**Fix:** add a one-shot migration that `TRIM`s `name` (and `test_script_name`) on existing rows in the six tables. The HMS team owns migrations per `hms-app/CLAUDE.md` ("tread carefully with migrations; consult peers"), so this should land as a separate PR after the app-level fix is in:

```sql
UPDATE "lab_sample"            SET "name"             = TRIM("name")             WHERE "name"             <> TRIM("name");
UPDATE "lab_collection_method" SET "name"             = TRIM("name")             WHERE "name"             <> TRIM("name");
UPDATE "lab_statuses"          SET "name"             = TRIM("name")             WHERE "name"             <> TRIM("name");
UPDATE "lab_unit"              SET "name"             = TRIM("name")             WHERE "name"             <> TRIM("name");
UPDATE "test_script"           SET "test_script_name" = TRIM("test_script_name") WHERE "test_script_name" <> TRIM("test_script_name");
UPDATE "service"               SET "name"             = TRIM("name")             WHERE "name"             <> TRIM("name");
```

Without this migration, the duplicate-detection fix is incomplete: a stale `"CBC "` row and a new `"CBC"` row can both exist and both be returned by list searches.

If a migration is out of scope, at minimum mention this in the PR description so the reviewer / ops team knows to schedule it.

## H3. Pattern concern: 6 hand-edited services, all repeating the same trim+check block — and the same bug almost certainly lives in cathlab / IPD / pharmacy

**Where:** all 6 files in this PR. Each one adds the same ~6 lines (`const trimmedName = payload.name.trim(); if (!trimmedName) throw ...`). The two patterns used to apply it (`data.name = trimmedName` vs `{ ...data, name: trimmedName }`) are inconsistent — three services mutate the assembled `data` object, three spread a new object. Both work, but it suggests copy-paste rather than a deliberate choice.

More importantly: a quick grep for other `*Service` classes with a `name` field (e.g. cathlab procedure types, IPD ward categories, pharmacy item categories) shows the same pattern of `findByName(payload.name)` without trimming. This bug is structural, not specific to the lab module. The fix as written addresses 6 instances; the codebase has more.

**Fix options, in order of preference:**

1. **Push it into the schema** (H1) — collapses the duplication, and any future `name` field gets the trim "for free" when copy-pasted from the example.
2. **Extract a small helper** — e.g. `requireTrimmedString(value, label)` in `utils/`, returning `{ value, error }`. The 6 services collapse to one line each. Lower payoff than (1) but works if the team wants to keep the schema thin.
3. **Open a follow-up ticket** to scan `*Service` classes for the same `findByName(payload.name)` pattern and apply (1) or (2) consistently. The ClickUp link in the PR body (`86exytyd7`) looks module-scoped, so this would be a separate epic.

At minimum the PR description should note "this is the lab module; the same pattern likely exists in cathlab/IPD/pharmacy and is tracked separately" so reviewers don't expect the fix to be exhaustive.

## M1 (minor). Two stylistic inconsistencies across the 6 services

- **Object construction style:** `lab-collection-method`, `lab-sample`, `lab-unit` build `const data = { name: trimmedName, ... }` and mutate. `lab-status`, `test-script`, `service` spread `{ ...data, name: trimmedName }` at the repository call site. Both are fine; pick one.
- **Empty-name error wording:** the 6 services use 6 different messages ("Collection method name cannot be empty", "Sample Type name cannot be empty", "Lab Status name cannot be empty", "Lab Unit name cannot be empty", "Test script name cannot be empty", "Service name cannot be empty"). Acceptable, but if H1 is taken the messages disappear and this becomes moot.
- **Stray blank-line removal** in `lab-collection-method.service.ts` (the diff drops a blank line after the `import { Prisma } from "@prisma/client";` line). Cosmetic; harmless but noisy in the diff.

## Things that look right

- **Coverage of both create and update paths.** All 6 services have both paths trimmed.
- **Duplicate check uses the trimmed value** in all 6 services — that's the actual bug fix.
- **HTTP status codes** — 400 for "name cannot be empty" matches the existing convention for the same services' "already exists" errors (`400` for lab-sample/lab-collection-method/lab-unit/service, `409` for lab-status/test-script — the inconsistency is pre-existing, not introduced here).
- **No `escapeSearchTerm` regression.** The earlier review flagged this; it has been removed.
- **No new tests added.** The trim + empty-check is straightforward enough that unit tests would add little, but a small test asserting `"   "` → 400 on one of the services would have been cheap insurance. Flagging as M2 (below) rather than blocking.

## M2 (minor, non-blocking). No tests for the new trim + empty-check behaviour

The PR doesn't touch `__tests__/` for any of the 6 services. Given that the trim is the entire reason for the PR, a single regression test per service asserting:

- `"  CBC  "` → stored as `"CBC"`, returns 200
- `"   "` → returns 400 with the "name cannot be empty" message

…would lock the behaviour in. Six short tests, no shared infra. Worth a follow-up if the team has a habit of covering lookup-table services; otherwise acceptable to ship without.

## Verdict

**Approve with minor suggestions.**

The trim is applied at the right layer (the service, which is also where the duplicate-by-name check lives — so the two stay in sync), covers create + update on all 6 services, and rejects whitespace-only names. The earlier `escapeSearchTerm` regression is fixed in this revision. The remaining findings are about polish (push the trim into the Zod schema, add a data backfill migration, open a follow-up ticket for other modules) rather than blocking correctness. The user-visible behaviour change — duplicate detection now works for whitespace-padded names — is achieved.

Suggested next steps for the author:

1. Lift the trim into the six Zod schemas (`z.string().trim().min(1, ...)`) and drop the duplicated guard blocks from the services.
2. Add a one-shot migration (separate PR, on `development` after this merges) that `TRIM`s the six columns for existing rows.
3. Either file a follow-up ClickUp ticket for cathlab/IPD/pharmacy or call it out in this PR's description as out-of-scope-but-tracked.
