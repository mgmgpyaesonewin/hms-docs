## Summary

This PR lands the HMS-side of the new CFI report feature: a new Prisma migration adding `event_outbox`, `cf_fee_reports`, and audit tables (with hand-rolled CHECK constraints and a `pg_trgm` GIN index); a new `src/lib/summary-api.ts` HMAC-less tenant-header HTTP client to talk to `127.0.0.1:4000`; a `enqueueCfFeeReportEvent` helper called from inside OPD billing transactions; a new UI page with table, pay/revert/activity-log/filter modals, columns and a sidebar entry; four new API routes under `src/app/api/(common)/reports/consultation-fees/`; a new `ConsultationFees` module in `permission-ui-config.ts`; and a new `.env.example` block.

The transaction discipline is correct (outbox INSERT happens inside the same `$transaction` as the OPD billing insert; the `Promise.all(serviceOperations)` is followed by the enqueue), and the Zod schemas + API response shape are coherent across the four new endpoints. But the PR has three issues that need to be addressed before merge: (1) **the `pay` and `revert` API routes do not perform any action-level permission check** — `verifyApiAuth()` only proves the user is logged in, and the new "Doctor Report / Consultation Fees" permission module does not define "Pay" or "Revert" actions, so any logged-in user can pay or revert invoices; (2) the new routes use the **older `apiHandler` + manual `verifyApiAuth()` pattern instead of the codebase's `enhancedApiHandler` (with `auth.permissions`)** that every other recent route uses; and (3) several TypeScript and React antipatterns slip through: a `useEffect` that intentionally omits `exhaustive-deps` (with an `eslint-disable`), a `throw new ApiResponse(...)` that doesn't extend `Error` and silently loses its stack trace, hardcoded 351-line table component without subcomponents, an outbox payload built by `JSON.stringify` on the server (Prisma `Json` values should be passed as objects), and a `WHERE next_attempt_at` partial index on `event_outbox` whose predicate (`status = 'PENDING'`) is not validated by the Prisma schema, making it easy to break with a future `prisma db push`.

## Issues

### Critical

- **`src/app/api/(common)/reports/consultation-fees/pay/route.ts:21` and `revert/route.ts:21` — No action-level permission check on write endpoints**
  Both routes call `verifyApiAuth()` (which only proves the user has a session) and then immediately call `summaryApi.post(...)`. The page UI gates "View Consultation Fees" via `PermissionGuard action="View" subject="Consultation Fees"` (`page.tsx:62`), but the new "Doctor Report / Consultation Fees" module in `permission-ui-config.ts:890-898` declares `permissions: crudExport` and `excludeActions: ["add", "edit", "delete"]`, leaving only `view` + `export` — there is no `pay` or `revert` action defined. Result: any logged-in user (including a doctor who only has "View" on the report) can hit `POST /api/reports/consultation-fees/pay` and pay arbitrary invoices, and any user can revert. The handler must call `checkPermission(session.user, "Pay", "Consultation Fees")` (or whatever the agreed action key is) before delegating to the summary service, and the permission module must define that action.
  Evidence: `pay/route.ts:21` — `const session = await verifyApiAuth();` — no `checkPermission` call; `permission-ui-config.ts` diff adds no `pay`/`revert` key.

- **`src/app/api/(common)/reports/consultation-fees/route.ts:13` — Wrong handler**
  `GET` uses the legacy `apiHandler` wrapper with a manual `await verifyApiAuth()` rather than the codebase's `enhancedApiHandler({ auth: { required: true, permissions: [{ action: "View", subject: "Consultation Fees" }] }, ... })`. There is no centralized place to enforce "Consultation Fees" `View` permission; a future refactor that swaps `verifyApiAuth` for something else will silently drop the check. The activity-logs route (`activity-logs/route.ts:13`) and `revert/route.ts:21` have the same problem. Fix: switch all routes to `enhancedApiHandler` mirroring `src/app/api/(appointment)/appointments/route.ts:6`.

### High

- **`src/app/(dashboard)/shared/opd/repositories/opd-billing.repository.ts:1286-1290` — Outbox event enqueued for non-consultation OPD bills**
  `enqueueCfFeeReportEvent(p, "cf_fee_report.bill_created", opdBill.id)` is called unconditionally after every `OPDBilling.create`, including pharmacy-only bills, lab bills, and imaging bills. The summary-service worker will then try to materialize `cf_fee_reports` rows for services that aren't consultation services. The event payload doesn't include the line items, so the worker has no way to filter. Either (a) gate the enqueue on `payload.services.some(s => s.isConsultation)`, or (b) include the filtered `opdBillingServiceIds` in the payload and let the worker skip non-consultation lines. This is the same problem on the `bill_reconciled` and `refund_created` events (`opd-refund.service.ts:67`).
  Evidence: `opd-billing.repository.ts:1286-1290` — no `services` filter; the event payload built in `cf-fee-report-events.ts:31` is `{ tenantId, opdBillingId }` with no `opdBillingServiceIds`.

- **`src/lib/cf-fee-report-events.ts:32-35` — Raw SQL insert with `JSON.stringify` payload + parameter typing issue**
  The `INSERT INTO event_outbox ... payload = ${payload}::jsonb` is passed via `tx.$executeRaw` tagged template, but `payload` is a string produced by `JSON.stringify`. Prisma's tagged-template `executeRaw` will correctly inline this as a string parameter, but the explicit `::jsonb` cast plus a JS-side `JSON.stringify` is a smell. More importantly, if the payload contains a string that breaks out of the JSON parsing on the worker side (e.g. a stray control character in a doctor's full name), Postgres will reject the cast. Recommendation: build the payload as a plain object and let Prisma serialize it: `INSERT INTO event_outbox (id, tenant_id, event_type, aggregate_id, payload) VALUES (gen_random_uuid(), ${tenantId}::uuid, ${eventType}, ${opdBillingId}::uuid, ${payload}::jsonb)` where `payload` is the object. Also: the `gen_random_uuid()` choice contradicts the schema's `@default(uuid(7))` — Prisma's `uuid(7)` is a TypeScript-side helper (calls `crypto.randomUUID()` in v5.18+), so the DB default and the Prisma default disagree. Since the PR uses raw SQL the DB default never fires, but the mismatch will trip up a future contributor who reads the schema and writes `tx.eventOutbox.create({...})` elsewhere.

- **`src/app/(dashboard)/common/reports/consultation-fees-report/features/api/cf-report.api.ts:18, 65, 92` — `throw new ApiResponse(...)` swallows the stack**
  `ApiResponse` (in `src/utils/api-response.ts`) is a plain class that does not extend `Error`. Throwing it produces a non-Error exception: the V8 stack trace is still captured, but the `error.name` is `"ApiResponse"` (not `"Error"`), and any `instanceof Error` check in a global error handler will be false, so it routes to a "500 Unknown error" path. Use `AppError` (which extends `Error` and carries an HTTP code) — see how `pay/route.ts:21` and `revert/route.ts:21` throw `AppError`. The `ApiResponse` class is meant to be the *response* shape, not an exception type.
  Evidence: `cf-report.api.ts:18` — `throw new ApiResponse({ ...data, statusCode: response.status });` — no `extends Error`.

- **`src/app/api/(common)/reports/consultation-fees/pay/route.ts:31-43` and `revert/route.ts:23-46` — Activity log writes silently fail, no user feedback**
  Both routes wrap `prisma.activityLog.createMany({...})` in `try { ... } catch (err) { logger.error("Failed to write CF pay activity logs", err); }`. The user sees "Consultation Fees Paid Successfully!" (the mutation already invalidated the query), but the activity log is missing for the row they just paid — and the next time they open the Activity Log modal, the row is empty. ### Medium

- **`src/app/(dashboard)/common/reports/consultation-fees-report/page.tsx:24-37` — `useEffect` with `eslint-disable react-hooks/exhaustive-deps` and `router.replace` race**
  The effect that back-fills the URL with today's date range on mount omits `searchParams`, `router`, and `effectiveQuery` from the dependency array. This works because it only runs once on mount (empty `[]`), but the eslint-disable hides a real concern: if a second navigation happens within the same mount (e.g. the user clicks a quick-range chip), the effect will not re-run. The right fix is to compute the next URL inside a `useMemo` and use `<Link>` / a single explicit `router.replace(effectiveUrl)` call in a click handler, or to derive `effectiveQuery` such that the back-fill is idempotent and remove the effect entirely. As-is, an ESLint run will surface the disabled rule again on every CI build until the effect is refactored.

- **`src/app/(dashboard)/common/reports/consultation-fees-report/features/components/consultation-fees-report-table.tsx:351 lines` — God component with mixed concerns**
  The table file mixes: (a) Mantine table rendering, (b) TanStack Table wiring, (c) Pay / Revert mutation orchestration, (d) bulk action bar (the floating `position: fixed` selection footer), (e) column-visibility menu, (f) export action formatting, and (g) `useEffect`-driven `barLeft`/`barWidth` measurement for the floating bar. The `useEffect` at line ~90 also has no cleanup beyond `removeEventListener("resize", update)` — a ResizeObserver would be more robust and survives font-load reflows (the floating bar's `left/width` is captured at mount but not when the table's content width changes). Split into `<CfReportTable>`, `<CfBulkActionBar>`, `<CfColumnVisibilityMenu>`, and a `useCfReportMutations` hook. Also: the 351-line file violates the project's "Keep files under 500 lines" rule (`/Users/pyaesonewin/CLAUDE.md` §Rules), so this is a hard cap hit waiting to happen.

- **`src/app/(dashboard)/common/reports/consultation-fees-report/features/components/consultation-fees-report-table.tsx` (selection bar `useEffect` + `barLeft/barWidth`) — Floating bar measures on mount but not on scroll**
  The fixed-position footer reads `containerRef.current.getBoundingClientRect()` on mount and on `resize` only. If the user scrolls so that the table's container scrolls under a sticky header / sidebar, the bar stays anchored to the viewport and no longer aligns with the table's horizontal edges. The fix is a `ResizeObserver` plus an IntersectionObserver on the container, or simply use `position: sticky; bottom: 0` and skip the JS measurement entirely.

- **`src/lib/summary-api.ts:30-37` — `fetch` has no timeout**
  A hung summary-service worker will block the request handler indefinitely (Next.js dev timeout is 30s, prod 60s; user-perceived timeout can be 60s+). Wrap the `fetch` in `AbortController` with a configurable `SUMMARY_API_TIMEOUT_MS` (default 10s) and `clearTimeout` in the catch path. The summary-service healthz probe should be the fallback, not a 60s wait.

- **`prisma/schema.prisma:6510-6620` — `CfFeeReport` lacks an FK to `OPDBilling` / `OPDBillingService`**
  The migration adds `opdBillingId` and `opdBillingServiceId` columns but no FK constraint to the source `opd_billing` / `opd_billing_services` tables. A stale `cf_fee_reports` row whose parent OPD bill has been hard-deleted will be orphaned. The summary-service worker uses these as the source of truth for "reconciled" diffs — if the OPD side deletes the bill but the cf_fee_reports row stays, the next `bill_reconciled` event will re-create a `cf_fee_reports` row pointing at nothing. Add `REFERENCES "opd_billing"("id") ON DELETE SET NULL` (or `CASCADE` if you trust the worker to handle cascades) in the migration, and add the `@relation` to the Prisma model.

- **`src/app/(dashboard)/common/reports/consultation-fees-report/features/components/cf-activity-log-modal.tsx:60` — `opened={!!row && !isLoading}` hides the modal during loading**

- **`src/app/(dashboard)/common/reports/consultation-fees-report/features/components/cf-report-columns.tsx:78-87` — `StatusBadge` renders two children without a wrapping `<Text>`**

## Verification needed

Things that cannot be verified from the diff alone and need a follow-up check:

1. **Does the migration apply cleanly to a fresh database?** The `prisma db pull` flow from the HMS-side schema (per `CLAUDE.md` §Cross-cutting) will not add the `pg_trgm` extension or the CHECK constraints — they live in this migration only. Verify on a clean dev DB: `npx prisma migrate deploy`.
2. **Does `enqueueCfFeeReportEvent` actually run inside the bill-created transaction?** The diff context shows `p` is the transaction client, but the test should be: kill the summary-service worker, create a bill, confirm the outbox row exists, then start the worker and confirm `cf_fee_reports` materializes. 3. **Does the floating selection bar align after a window resize and a content-width change?** Manual QA in browser, with both desktop and tablet viewports.
4. **Does the activity-log modal stay usable when the user clicks through 5+ rows quickly?** There is no in-flight cancellation; React Query will fire 5 queries, and the modal's `opened={!!row && !isLoading}` race could leave a stale `logs` array shown. The user should verify whether the modal swaps mid-flight are visible.
5. **Does the permission module render in the role-edit UI?** The new "Doctor Report / Consultation Fees" submodule uses `crudExport` + `excludeActions: ["add","edit","delete"]` — confirm that the role editor's action checkboxes correctly hide the excluded ones and that "View" + "Export" are the only available actions.
6. **Do pay and revert work end-to-end against the summary-service worker?** The summary-service in the local dev compose (per `infra/docker-compose.yml`) must be running and pointing at the same Postgres. The new `.env.example` `SUMMARY_API_URL=http://summary-api:4000` is the docker-compose service name — verify it resolves to the summary-api container.
7. **Does the activity log row actually get created on the HMS side when the worker succeeds?** The high-priority issue above (silent catch) means this is currently a best-effort write. Manual test: pay one row, then re-open its activity log — if it's empty, the catch is masking a real Prisma error.

## Recommendation

Block merge until the three real blockers are fixed: (1) add a `checkPermission` call (or `enhancedApiHandler` with `auth.permissions`) on the `pay`, `revert`, `activity-logs`, and `list` API routes and define the corresponding "Pay" / "Revert" actions in `permission-ui-config.ts`; (2) switch the activity-log write from a best-effort `createMany` with `try/catch` to a single source of truth (either the summary-service audit tables or the HMS activity log, not both); and (3) fix the activity-log modal's loading-state race by keeping the modal mounted during `isLoading`. The other issues (table size, `useEffect` cleanup, fetch timeout, missing FK to OPD billing) are worth addressing in this PR or a follow-up before any production rollout — the `summary-api.ts` HMAC removal is acceptable for v1 dev only and must be re-evaluated before the on-prem deployment.
