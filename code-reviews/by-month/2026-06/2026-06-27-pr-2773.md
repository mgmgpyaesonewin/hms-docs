## Summary

The fix targets a real bug: in the ED EMR tab, once an `OPDBilling` invoice exists for the appointment, the ED-bill update action unconditionally throws "Cannot modify emergency services after an invoice is generated." That blanket rejection blocks the legitimate "cancel a service / procedure / pharmacy item" workflow, which the EMR-side UI still supports (it just disables every other field). The PR narrows the rejection so that updates after an invoice are permitted **only** when the diff is restricted to cancel flips on existing items (and headers / package membership are byte-for-byte identical).

The main change lives in `ed-emr-emergency-services.actions.ts`: a new `assertCancelOnlyEdBillUpdate` compares the incoming payload against the persisted bill (`existingBill.serviceBill`, `existingBill.procedureBill`, `existingBill.pharmacySale`, `existingBill.servicePackageBill`) and throws `AppError` if anything other than a cancel transition changes. Supporting changes plumb the missing fields (`localPrice` / `foreignerPrice` / `insurancePrice` / `governmentPrice` / `discountAmount` / `discountPercentage`) into the procedure form mapping (`use-bind-form.tsx`), reshape the `EdBillingProcedureType.procedure` shape to flat price numbers (`emergency-billing.types.ts`), and forward an `hasOpdSlip` prop computed from `edBillingData.appointment.OPDBilling.length` to `ProxyBillProcedures` so the procedure row disables itself when an OPD slip exists (parallel to the existing `hasOpdBillingInvoiceId` restriction).

The implementation is mostly correct and self-aware — the inline comment correctly notes that procedures don't carry a stable id through the form and matches them by field values instead. But there are two real correctness gaps and a couple of cleanups before this should ship: (1) the service-items match is done by `item.id` even though the repository on update does `serviceBillItems: { deleteMany: {}; createMany: { ... } }` (new ids), so a service add/remove/delete sequence that preserves the visible *order* will pass the assertion but still mutate state (the assertion only blocks changes, it doesn't fully reconstruct equivalence); (2) `assertCancelOnlyEdBillUpdate` is reached only from `updateEdBillFromEmrAction`, but the *create* paths (`createEdBillFromEmrAction`, `createStandaloneEdBillAction`) keep the unconditional "Cannot modify after invoice" throw — so if a user opens the EMR form, the ED bill already exists (from a prior save), and they click "Add Service" the form re-issues as an *update*, but if they click "Save" without re-loading and the form re-fires `createEdBillFromEmrAction` somehow, they will hit the old throw; (3) the new `hasOpdSlip` plumbing in `ed-billing-procedure.tsx` derives the boolean from `OPDBilling.length`, which counts slips regardless of payment/cancel status — a fully cancelled OPD slip will still lock the form.

## Issues

### Critical

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:178-209` — Service-items match-by-id is wrong: the repository reassigns ids on every update**
  The repository's `updateEdBill` runs `serviceBillItems: { deleteMany: {}; createMany: { ... } }` (`ed-bill.repository.ts:470-491`), which **mints new row ids** for every service item on every save. So the *persisted* ids in `existingBill.serviceBill.serviceBillItems` after a save will not match the ids that were just sent in the payload. The new assertion (`actions.ts:185-192`) does `item.id && !usedServiceIds.has(item.id) ? existingServices.find((s) => s.id === item.id) : undefined` — but `item.id` is the *new* id from the form, while `existing.id` is the *old* id from the database. The two id spaces are disjoint for any save past the first.
  Concretely: user opens the ED EMR tab → form hydrates with existing services, each carrying its persisted `id` (`use-bind-form.tsx:33: id: itm.id`). User cancels one service (the only allowed change) and saves. The payload still carries the **persisted** ids (because `useBindForm` only re-maps on `edBillingData` change). On this single re-save path, the assertion happens to work. But on any later edit, `useBindForm` is not re-run (no `edBillingData` change), the form keeps carrying the now-stale ids, and the assertion at `actions.ts:185-186` will fall through to `existingServices.find(...)` and either (a) silently match nothing because the stale id is gone, or (b) collide with the wrong row if the ids were ever recycled. The correct fix is to match service items by `(serviceId, doctorId, priceType)`-tuple rather than by id, just like procedures — or assert that *every* payload item must find a match in the existing set by tuple identity and then compare only the cancel flag.
  Evidence: `actions.ts:185-186` — `item.id && !usedServiceIds.has(item.id) ? existingServices.find((s) => s.id === item.id) : undefined;` — `item.id` is stale after a prior save because `ed-bill.repository.ts:470` deletes and re-creates.
  Fix: replace the id-based lookup with `(s) => valuesMatch(s.serviceId, item.serviceId) && valuesMatch(s.doctorId, item.doctorId) && valuesMatch(s.priceType, item.priceType)`, and use a `Set` of used tuples (or an in-place splice) to enforce 1-to-1 mapping, the same pattern used for procedures at `actions.ts:200-218`.

### High

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:111-114, 54-59` — `createEdBillFromEmrAction` and `createStandaloneEdBillAction` still throw unconditionally**
  The bug ticket ("ED emr cancel service error") is fundamentally about the *update* path, but the *create* paths (`createEdBillFromEmrAction` at `actions.ts:109-115` and `createStandaloneEdBillAction` at `actions.ts:54-59`) still keep the old `if (existingBillCount > 0) throw new AppError(...)` block.   Evidence: `actions.ts:54-59` — same throw as before; `actions.ts:111-114` — same throw as before; only the update branch at `actions.ts:135-140` is replaced.

- **`src/app/(dashboard)/ed/features/components/ed-billing-procedure.tsx:13` — `hasOpdSlip` counts all OPD slips, including `CANCEL`-status ones**
  `const hasOpdSlip = Boolean(edBillingData?.appointment?.OPDBilling?.length);` includes cancelled OPD bills, while the existing `existingBillCount` block in the action (`actions.ts:131: opdBillingPaymentStatus: { not: "CANCEL" }`) explicitly excludes them. The two checks now disagree: the action permits updates when only a CANCEL OPD bill exists, but the UI locks the procedure add-button when *any* OPD bill exists. This produces the inverse of the bug the PR is fixing — the user cannot add a new procedure on an appointment whose OPD slip was voided, even though the action would accept the save.
  Fix: mirror the action's filter. Either expose a `slipCount` (cancelled + non-cancelled) and a `paidSlipCount` (non-cancelled) from the backend, or filter client-side if the OPD billing type carries a status field — `edBillingData?.appointment?.OPDBilling?.some(b => b.status !== "CANCEL")`.
  Evidence: `ed-billing-procedure.tsx:13` — `Boolean(edBillingData?.appointment?.OPDBilling?.length)` — no status filter; `actions.ts:131` — `opdBillingPaymentStatus: { not: "CANCEL" }`.

### Medium

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:147-176` — `existingBill.displayFOCField` vs `payload.displayFOCField` boolean coercion asymmetry**
  Line 162: `Boolean(existingBill.displayFOCField) !== Boolean(payload.displayFOCField)`. The schema (`ed-bill.schema.ts:294`) coerces the inbound value with `z.coerce.boolean().default(false)`, so `payload.displayFOCField` is already a real boolean — the extra `Boolean(...)` is redundant. `existingBill.displayFOCField` comes from the DB and is nullable (Prisma's `Boolean?`). Using `Boolean(...)` on both sides is fine, but the asymmetry hides a real concern: if the DB row has `displayFOCField = null`, `Boolean(null)` is `false`, and the assertion will reject a payload that explicitly sets `displayFOCField = true`. Consider using `existingBill.displayFOCField ?? false` instead so an undefined DB value does not silently equal `false`.
  Evidence: `actions.ts:162` — `Boolean(existingBill.displayFOCField) !== Boolean(payload.displayFOCField)`.

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:163-164` — Date comparison via `new Date(...).getTime()` swallows time-of-day differences**
  `new Date(existingBill.date).getTime() !== new Date(payload.date).getTime()` will return `false` (i.e. match) only if the milliseconds are equal. Both sides will round to the second after a JSON round-trip, but the form posts `date: dayjs(edBillingData.date).toDate()` (`use-bind-form.tsx:194`) which carries millisecond precision, while the DB stores `date` as a Prisma `DateTime` (microsecond precision in Postgres). The comparison will frequently fail on the very first re-save even when nothing changed, blocking legitimate cancel-only updates.
  Fix: compare at second or minute resolution (`Math.floor(t/1000)`, or strip the time component entirely with `dayjs().startOf("minute")`), or compare only the calendar day (`YYYY-MM-DD`).
  Evidence: `actions.ts:163-164` — `new Date(existingBill.date).getTime() !== new Date(payload.date).getTime()` — both are full timestamps.

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:225-236` — Package set comparison uses `Array.some(id, idx => id !== payloadPackageIds[idx])` instead of deep equality**
  Lines 232-236: the `existingPackageIds.length !== payloadPackageIds.length || existingPackageIds.some((id, idx) => id !== payloadPackageIds[idx])` is correct *because* both arrays are pre-sorted, but it relies on the reader seeing the `.sort()` at lines 230 and 232. A safer pattern is to compute a `Set` once and compare `existing.size === payload.size && [...existing].every(id => payload.has(id))`. Stylistic, but the current form invites a future contributor to drop a `.sort()` and ship a silent bug.
  Evidence: `actions.ts:232-236`.

### Low / Nit

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:127-131` — `valuesMatch` / `amountsMatch` could be one helper**
  Two near-identical helpers (`valuesMatch` for null-safe `===`, `amountsMatch` for null-safe `Number(...) === Number(...)`). The only practical difference is the coercion; the code would be shorter as `equalLoose(a, b) = (a ?? null) === (b ?? null) || Number(a ?? 0) === Number(b ?? 0)` with the caller deciding which mode to apply. Or, since `Number(undefined)` is `NaN` and `NaN !== NaN`, the second helper has a quiet footgun when both sides are `undefined` (which the `?? null` saves you from here, but a future refactor might not).
  Evidence: `actions.ts:127-131`.

- **`src/app/(dashboard)/shared/ed/types/emergency-billing.types.ts:135-140` — `EdBillingProcedureType.procedure.items[]` removed, but a downstream consumer may still rely on it**

- **`src/app/(dashboard)/ed/features/hooks/use-bind-form.tsx:103-106` — The four procedure prices are added to the payload but the schema's `ProcedureBillItemSchema` already declares them (`ed-bill.schema.ts:158-173`)**

Run `npm run typecheck` and grep `ProxyBillProcedureBillSchema` to confirm.
  Evidence: `use-bind-form.tsx:12` — imports `ProxyBillProcedureBillSchema` from a different schema path.

- **`src/app/(dashboard)/emr/ed/features/emergency-services/ed-emr-emergency-services.actions.ts:127-131` — `amountsMatch` falls back to `Number(...)` which silently converts strings**
  `Number("123abc")` is `NaN`, which is not equal to `NaN`, so a stringified bad value will appear to "not match" and the assertion will throw — masking the underlying data corruption with the "cannot modify" error. Acceptable for this fix, but worth a one-line comment in `amountsMatch` explaining the `NaN` behavior so the next person doesn't try to "fix" it.
