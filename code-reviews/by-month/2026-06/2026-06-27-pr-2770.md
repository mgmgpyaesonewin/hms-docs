## Summary

This PR lands two related IPD fixes. The headline change is in `src/app/(dashboard)/ipd/room/request-room/features/components/room-log-form.tsx`: when editing a room log whose admission is no longer in the active-admissions list (e.g. discharged, or an older record on a `REQUESTED` bed), the form now includes that admission in the patient `Select` options and resolves the patient id from the room-log data as a fallback. A secondary bug-fix corrects a route typo in `get-room-log-by-room-id.api.ts` from `/api/room-lod/${id}` to `/api/room-log/${id}`, and `room-card.tsx` is hardened so the patient falls back to `activeRoomLog.admission?.patient` when `activeRoomLog.patient` is null.

The IPD-side feature work adds a CT-specific ward-clearance block. A new repository method `DischargeRepository.countUnbilledAcknowledgedCtAddOns` counts acknowledged CT services with unposted add-on records, and `DischargeService.updateDischargeWardClearanceStatus` calls it before the existing `findPendingWardRequestsForDischarge` check and throws a specific `AppError("Post CT Error: …", 400)` when the count is non-zero. The CT condition is also added to `findIncompleteRequestsForDischarge` and to the `OR` clause in `findPendingWardRequestsForDischarge` so the generic pending-service counter still catches unbilled CT add-ons as a defensive aggregate. Tests are added at the repository, service, and component level (a new `__tests__/` directory collects previously colocated `__tests__/` files and adds three new spec files). A design doc is committed at `docs/superpowers/specs/2026-06-22-ct-ward-clearance-alert-design.md`.

The implementation is correct and well-tested, but there are several issues to address before merge: (1) the new repository method `countUnbilledAcknowledgedCtAddOns` runs the CT query, then `findPendingWardRequestsForDischarge` runs **another** duplicate CT-count query (because the CT condition was added to its `OR` clause), so the ward-clearance path now makes two identical round-trips for every CT-bearing admission; (2) the room-log-form changes are **not** covered by a test that exercises the "admission not in active list" branch — the new test only asserts that the requested admission's patient shows up in the select, but never asserts the fallback path when the admission is missing from `admittedPatientData`; (3) the design doc references a stray `p` property on `findIncompleteRequestsForDischarge` and a spec note about "enum consolidation" that don't reflect the actual code as shipped; (4) the new `__tests__/` directory duplicates paths that already exist under `src/.../services/__tests__/` and the new repo tests are tested with `prisma as unknown as { … count: jest.fn() }` — pure mocks, not real Prisma, so the typed-filter changes are not actually verified against the generated client.

## Issues

### Critical

- **`src/app/(dashboard)/shared/ipd/repositories/discharge.repository.ts:762-781` and `:839-855` — Duplicate CT count query on every ward-clearance transition**
  `findIncompleteRequestsForDischarge` adds a CT `OR` branch (lines 762-781) and `findPendingWardRequestsForDischarge` adds the **same** CT condition (lines 839-855). On the ward-clearance path (`discharge.service.ts:680-693` BEFORE state), the flow is now: (a) `countUnbilledAcknowledgedCtAddOns` (line 678 insert) → (b) `findPendingWardRequestsForDischarge` (line 681) which internally re-counts the same CT set. Two CT `iPDDailyService.count` round-trips fire for every ward-clearance transition that has any CT imaging. The first one (`countUnbilledAcknowledgedCtAddOns`) is needed to select the user-facing error message; the second one is duplicate work that adds latency and concurrency risk without changing behavior.
  Fix: remove the CT branch from `findPendingWardRequestsForDischarge` (`discharge.repository.ts:839-855`) so the aggregate counter focuses on the non-CT pending requests. The dedicated `countUnbilledAcknowledgedCtAddOns` method is the single source of truth for the CT check; the service already short-circuits before calling the aggregate when the CT count is non-zero (see the `discharge.service.node.test.ts:198-202` test which asserts `findPendingWardRequestsForDischarge` is **not** called when the CT count is non-zero). Or, equivalently, compute the CT count once at the start of the service method and pass it to both consumers. The cleaner option is removal of the duplicate `OR` branch.
  Evidence: `discharge.repository.ts:839-855` — full duplicate of the `OR` clause at `:762-781`; `discharge.service.ts:678-693` — sequential `await` calls fire both queries.

### High

- **`src/app/(dashboard)/ipd/room/request-room/features/components/room-log-form.tsx:74-118` — New "admission not in active list" branch is untested**
  The diff adds a fallback in `patientWithAdmissionIdOpts` (`room-log-form.tsx:74-95` of the patched file) that injects the current room log's admission into the options when `admittedPatientData?.result.admissions` does not contain it (e.g. discharged). It also adds a fallback in `getAdmissionPatientId` (`room-log-form.tsx:101-110`) that resolves from `roomLogData?.result.roomLogs` when the active-admissions lookup misses. The new test (`__tests__/requested-room-log-form.test.ts`) mocks **both** queries to include the requested admission in `admissions` with `status: "ACTIVE"`, so the fallback branches are never exercised. The test asserts the rendered label is correct, but if the implementation silently dropped the fallback (regression risk), the test would still pass because the mock supplies the data through the primary path. Add a second `it("binds the admitted patient when the admission is not in the active list", ...)` that mocks `admittedPatientData?.result.admissions` to omit the requested admission (status `DISCHARGED` or absent entirely) and asserts the same label is still rendered via the fallback.
  Evidence: `__tests__/requested-room-log-form.test.ts:497-526` — the `admissions` mock includes both `oldAdmissionId` and `requestedAdmissionId` with `status: "ACTIVE"`, so the `admissionFromRoomLog` fallback in `room-log-form.tsx:78-90` never triggers.

- **`src/app/(dashboard)/shared/ipd/services/discharge.service.ts:678-693` — Two sequential `await`s on independent counts; the second one may execute after the first throws and is not parallelized**
  In the patched file, `countUnbilledAcknowledgedCtAddOns` is awaited first, and only on success is `findPendingWardRequestsForDischarge` awaited. The two counts are independent — both filter on `discharge.admissionId` and could run in parallel via `Promise.all`. As-is, the worst-case latency is `t1 + t2`. If both queries target indexes correctly (the `addOn` query is on a `?` relation, the pending-ward query is on multiple `count`s in a `Promise.all`), the `Promise.all` pattern at the repository level would not compose cleanly, so this is more of a code-organization issue than a perf hot-spot. Still: in a high-traffic IPD ward, ward-clearance transitions are bursty (e.g. end-of-shift), and serial awaits compound. Recommend `const [ctCount, pendingWardRequests] = await Promise.all([...])` if/when both queries are needed (i.e. after the duplicate-CT branch from the Critical issue is removed). With the Critical fix in place, this becomes a Medium-priority polish.
  Evidence: `discharge.service.ts:678-693` of the patched file — `await this.dischargeRepository.countUnbilledAcknowledgedCtAddOns(...)` then `await this.dischargeRepository.findPendingWardRequestsForDischarge(...)`.

### Low

- **`src/app/(dashboard)/shared/ipd/repositories/discharge.repository.ts:843-860` — `countUnbilledAcknowledgedCtAddOns` does not parallelize with other counts and uses `prisma.iPDDailyService.count` even though `prisma.iPDDailyService` is the related relation**
  The query joins through `dailyBill: { admissionId }` and `ImagingService: { type, status, addOn: { is: { OR } } }`. The `addOn.is` filter forces a join to `ImagingServiceAddOn`, but the existing schema (`imaging_service_add_ons.imaging_service_id @unique`) means at most one add-on per imaging service. A simpler and faster query would be `prisma.imagingServiceAddOn.count({ where: { isCharged: false, chargedIpdDailyBillId: null, imagingService: { type: "CT", status: "ACKNOWLEDGED", ipdDailyService: { dailyBill: { admissionId } } } } })`. The current query is correct but routes through `iPDDailyService` which is a denormalized join path. Worth a future perf ticket if CT-clearance starts to feel slow in production.

If `operationTypeIds = ["operation-2", "operation-1"]` and `operationTypes` is in `id`-sorted order, the Map iteration order is `operation-1` first (insertion order from `.map(...)`), but the output is built from `operationTypeIds.map(...)` so the request order is preserved. Add a test for the partial-match case to lock the current behavior (or change the behavior to surface a console warning).
