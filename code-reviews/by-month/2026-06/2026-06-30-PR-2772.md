## Summary
Default-sorts the "Appointments by Doctor" report table by `totalAppointments` desc via `useMemo`. The two other changed entries are submodule pointer bumps for `appointment` and `opd`.

## Findings

### Important
- `appointments-by-doctor-table.tsx:40-45` — `[...data].sort(...)` mutates a *copy* on every render where `data` changes, but sorting belongs in the data layer (server-side `orderBy` on the fetch query) so the API contract already returns sorted rows and the UI doesn't pay the sort on every render. If the upstream `data` already arrives paginated, server-side ordering also keeps the sort stable across pages. Verify the fetch in `appointments-by-doctor.api.*` accepts an `orderBy`; if not, this is fine as a local fallback, but it should be documented in the comment rather than just `// Default sort: total appointments descending`.
- `appointments-by-doctor-table.tsx:42` — `useMemo` here is a premature optimization for what is at most a few-hundred-row report. The `DataTable` likely already memoizes rows; adding a `useMemo` for a pure `.sort()` adds dependency tracking overhead without measurable benefit. One line removed (`useMemo` wrapper), inline the sort — `[...data].sort(...)` is already O(n log n) and effectively free at this size.

### Nit
- `appointments-by-doctor-table.tsx:42` — `data` is typed such that `data` may be `undefined`; the `if (!data) return []` guard is correct, but the cast `[] as AppointmentsByDoctorReportSchema[]` is unnecessary if `data`'s type already includes `undefined` — `[]` alone satisfies the consumer. Drop the cast.

## Verdict
Approve with suggestions
Trivial change, functionally correct, but the sort should live in the API query if possible; `useMemo` is overkill for the row counts involved.
