## H1. `escapeSearchTerm` is now applied to values that get persisted, corrupting user-typed LIKE-special characters in 4 tables that did not have this problem before

**Where:** `lab-sample.service.ts:25-26,41,49-50`, `lab-collection-method.service.ts:33-34,42,55-56`, `lab-status.service.ts:32-33,45-48,82-87`, `test-script.service.ts:31-32,44,76-79`

The pattern in the PR is `safeName = escapeSearchTerm(trimmedName)`, and `safeName` is then **saved** to the DB (`data.name = safeName` or `{ ...data, name: safeName }`).

`escapeSearchTerm` is a SQL-LIKE escaper:

```ts
// src/utils/general-utils.ts:7
export function escapeSearchTerm(term: string): string {
  return term.replace(/[%_'"\;\\]/g, "\\$&");
}
```

It is only meaningful when the value is consumed by a `LIKE` query — the backslashes tell the DB to treat `%`, `_`, etc. as literals. Prisma's `equals` filter is **not** a LIKE comparison, so the function adds literal backslashes to the value that get stored as-is. (The same convention already lives in `service.repository.ts:208` and `lab-unit.repository.ts:63` — so the bug is older, but the PR extends it to four more tables that did not have it.)

Net effect for the four tables newly affected by the PR:

- A user types `"100%"` as a sample / collection method / status / test-script name.
- The DB now stores the literal string `"100\%"`.
- The list view and detail view render the name with the backslash in it.
- Anywhere else in the codebase that calls `prisma.labSample.findFirst({ where: { name: { equals: <user input> } } })` with the raw user input will miss the record.

The "no-op escape on `equals`" pattern is pre-existing in the codebase, so this is consistent with house style — but the PR doubles the blast radius (4 new tables), so it is worth flagging explicitly.

**Fix:** apply the trim and store the trimmed (unescaped) value, and only escape at the call sites that actually do `contains` / `startsWith` / `endsWith`:

```ts
const trimmedName = payload.name.trim();
if (!trimmedName) {
  throw new AppError("Sample Type name cannot be empty", 400);
}
const data = { name: trimmedName, ... };
const existing = await this.labSampleRepository.findLabSampleByName(trimmedName);
```

`escapeSearchTerm` should only be used inside repository methods that build a `contains`/`startsWith`/`endsWith` filter, never on values stored via `equals` or written to a column.

## H1. List endpoints in the 4 tables do not escape the search term, so newly escaped stored values are no longer findable by user search

**Where:**
- `lab-sample.repository.ts:97-102` (`contains: query.search` — no escape)
- `lab-collection-method.repository.ts:103-110` (`contains: query.search` — no escape)
- `lab-status.repository.ts:18-23` (`contains: query.search` — no escape)
- `test-script.repository.ts:18-23` (`contains: query.search` — no escape)

Before this PR, the values stored in these tables did not contain backslashes (the repositories' `equals: name` passed the raw input straight through, and the search was unescaped on both sides). After the PR, newly created rows store values like `"100\%"`, but the list search still does `contains: query.search` against the user-typed `100%` — so the user can no longer find the row they just created by typing its name. Conversely, a user searching for a name that contains a `%` or `_` will get a wildcard-matched result set.

This is a regression introduced by the PR for these 4 tables (the lab-unit and service tables had it pre-existing).

**Fix:** at the very least, escape the search term in the `contains` filters so it is consistent with the stored value:

```ts
where.name = {
  contains: escapeSearchTerm(query.search.trim()),
  mode: "insensitive",
};
```

(Or, if finding H1 is taken, fix it at the source by storing unescaped trimmed values and only escaping at the `contains` call site — then this finding goes away.)

## H2. No migration for existing rows that have leading/trailing whitespace — the new duplicate check is blind to them

**Where:** all 6 services. The duplicate check is now `findByName(safeName)` where `safeName` is trimmed. Existing rows in the DB that were inserted before this PR (e.g. `"CBC "` with a trailing space) are not visible to the new check, because the check looks for the trimmed value `"CBC"`.

Concrete consequences:

1. A user creates `"CBC "` in production today. The PR is deployed. A new user tries to create `"CBC"` — the lookup is `findByName("CBC")`, which does not match `"CBC "` (Prisma `equals` is byte-exact), so creation succeeds. Now there are two rows that are semantically identical.
2. A user edits `"CBC "` → submits `"CBC"`. The duplicate check looks for `"CBC"`, finds nothing, and renames the row to `"CBC"`. Fine, but step (1) above shows the inverse direction is also possible and is the bug.
3. The trim is one-way: once an existing padded row exists, it cannot be detected as a duplicate of a future trimmed row, but it can block creation of a future row that has the same name with different padding.

**Fix:** add a one-shot migration that trims `name` (and `testScriptName`) on existing rows in these 6 tables, and rely on the application-level check to keep them clean going forward. Trimming before the new code ships is the only way to avoid the pre-existing padded rows being "stranded" as invisible duplicates.

A small SQL migration in `prisma/migrations/` is sufficient — for example:

```sql
UPDATE "lab_sample" SET "name" = TRIM("name") WHERE "name" <> TRIM("name");
UPDATE "lab_collection_method" SET "name" = TRIM("name") WHERE "name" <> TRIM("name");
UPDATE "lab_statuses" SET "name" = TRIM("name") WHERE "name" <> TRIM("name");
UPDATE "lab_unit" SET "name" = TRIM("name") WHERE "name" <> TRIM("name");
UPDATE "test_script" SET "test_script_name" = TRIM("test_script_name") WHERE "test_script_name" <> TRIM("test_script_name");
UPDATE "service" SET "name" = TRIM("name") WHERE "name" <> TRIM("name");
```
