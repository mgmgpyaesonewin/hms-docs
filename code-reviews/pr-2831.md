# PR #2831 — fix(room transfer): invalidate additional queries for room transfer functionality

**Author:** Paing Sett Kyaw (@DaDDy-chilll) **Branch:** unknown **Files changed:** 3 (+194 / -1)

## Summary
After a successful room transfer, invalidate `building-with-rooms`, `room-type-by-ward-id`, and `occupied-admission-patients` queries. Also filter the "transfer-to" room dropdown to hide rooms already occupied per the occupied-patient feed.

## Findings

### Critical
None.

### Important
None.

### Nit
- `room-transfer-form.tsx:405-413` — `occupiedTransferToRoomIds` is recomputed on every render. With `occupiedAdmissionPatients` rarely changing while the modal is open, a `useMemo` is the minimum-lazy guard if the list grows past ~tens of rooms; today it's fine. Marking only so future growth doesn't surprise.
- `room-transfer-form.tsx:412` — filter assumes the occupied-patient feed is the single source of "occupied" rooms for the dropdown. If a room becomes occupied via a code path that doesn't write to `occupied-admission-patients` (e.g., direct admission), it would still appear selectable. Worth a one-line comment on that invariant, or route the filter through the existing `building-with-rooms` data. (`housekeeping-modal.tsx:116` and `edit-room-log-modal.tsx:33` already invalidate the same key, so the two lists will eventually converge — just don't rely solely on it.)

## Verdict
Approve.
Query-key additions match existing patterns in `housekeeping-modal.tsx` / `edit-room-log-modal.tsx`. Dropdown filter change is local, tested, and uses data already on screen.
