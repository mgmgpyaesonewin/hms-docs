## Summary
This is the fifth in a series of "fee reports" PRs (after TC, In-House Doctor, Procedure, Reading) that materializes a `rf_fee_reports` summary table from the transactional outbox emitted by the HMS. It introduces a generic, report-agnostic event-emission helper pair (`opd-billing-events.ts`, `ipd-daily-bill-events.ts`) and wires them into every IPD/OPD mutation surface (cathlab, ct-add-on billing, ward service, service request, daily bill reconcile, proxy/team fee, OPD cancel/delete/refund, OPD bill update) so the worker can fan out one event into N reports. The HMS-side adds a shared `summary-api.ts` HTTP client, a Round Fees report page mirroring the four prior reports (table + filter/pay/revert/activity-log modals + three Next.js route handlers proxying to the summary service), Prisma schema additions + migration with CHECK constraints and a pg_trgm GIN index for `invoice_no` search, sidebar and permission-config entries, and small touches to `cathlab.service`, `ward-service.service`, `service-request.service`, `ipd-daily-bill.service`, `ct-add-on-billing.service`, `opd-billing.repository`, `proxy-bill-template.service`, and `opd-refund.service`.

The series is now mature enough that the structural pattern is clearly settled: one migration per report, one report-table UI per fee type, four routes per report (list, pay, revert, activity-logs), one generic event-emit helper per source domain. Most of the volume here is repetition of the settled pattern rather than new architecture — which is what the rubric below rewards/penalizes.

## Issues

### Critical

**C1. Server-to-service auth was removed without a compensating control.** `src/lib/summary-api.ts` (new file, 66 lines) sends only `X-Tenant-Id` to the summary-api and ships a comment explicitly admitting "the header is trusted on the wire (both services are on the same private network)." The summary-service `tenant-guard` middleware then trusts that header to scope every Prisma query. This is a defense-in-depth bypass: the HMAC + replay-cache + ±5-min-skew machinery documented in `CLAUDE.md` (ADR 0008) and `hms-docs/summary-service/api/hmac-auth.md` is gone, and the comment frames it as "v2 will reintroduce." The HMS service is publicly reachable at `http://localhost:3000`; any code path or compromised dependency that reaches `src/lib/summary-api.ts` can now forge a `DEFAULT_TENANT_ID` and read/mutate every CFI/IH/PF/RF row for any tenant the worker knows about. The summary-api still binds to `127.0.0.1` per `CLAUDE.md`, which limits external reachability, but that is not a control — it's a network accident. Either restore HMAC on the v1 wire (cheap, already designed) or write the rationale as a `CONTEXT.md` ADR with an explicit "trusted network only" runbook constraint, and add a regression test that fails if HMAC middleware is missing.

### High

**H1. `enqueueOpdBillingEvent` / `enqueueIpdDailyBillEvent` silently drop events when `DEFAULT_TENANT_ID` is unset.** Both helpers (new `src/lib/opd-billing-events.ts` L36-41 and `src/lib/ipd-daily-bill-events.ts` L30-35) log a warning and `return` without writing to `event_outbox` if the env var is missing. Every other caller in this PR wraps the emit in `prisma.$transaction` (see `opd-billing.repository.ts:7711`, `:7755`, `:7780`, `:7802`, `:7816`, and the cathlab / ward / IPD daily-bill / proxy-bill additions). The transaction commits the user-visible write but the outbox row never lands, so report rows silently drift out of sync with the bill. The only signal is a `winstonLogger.warn`. In production this is exactly the kind of bug that manifests as "the round-fee report doesn't show today's procedures" with no error path. Either throw inside the transaction (so the user's bill write rolls back — the right answer for a "data must stay in sync" invariant), or fail-fast on app boot when the env var is missing. "Log and continue" is a foot-gun for an outbox.

**H2. `last_synced_at DEFAULT CURRENT_TIMESTAMP` plus a unique-on-event-id-index mismatch will silently break idempotency.** The new `rf_fee_reports` migration (L675-677) declares `last_synced_at` as `TIMESTAMPTZ(6) NOT NULL DEFAULT CURRENT_TIMESTAMP` and is paired with five `UNIQUE` partial-style indexes — one per source-grain column (`opd_billing_service_id`, `ipd_daily_service_id`, `ipd_ward_service_item_id`, `ipd_service_bill_item_id`, `ipd_cath_lab_service_item_id`). The accompanying comment ("Nullable unique: Postgres treats NULLs as distinct") describes Postgres semantics correctly, but does not check the corresponding five UNIQUE constraints on the *earlier* migrations (TC / IHD / PF). When this series ships, every report type has the same nullable-unique-with-grain-columns pattern; if any one of the five reports has a duplicate-grain-id collision (e.g., a single IPD daily-service line legitimately maps to two fee categories), the unique constraint blocks the worker's `INSERT … ON CONFLICT` and the whole reconciliation silently dead-letters. Audit `prisma/migrations/20260618090000_add_tc_fee_report_tables`, `20260623120000_add_ihd_fee_report_tables`, `20260626050919_add_pf_fee_report_tables`, and `20260629000000_add_rf_fee_report_tables` against the worker code on the summary-service side and prove there is no row whose grain columns could both be non-null at the same time. If there is, the migration as written will fail at runtime, not at migration time.

**H3. `activityLogger.createMany` is wrapped in a `try/catch` that downgrades an audit failure to a soft warning on the response.** Each of the five `pay` / `revert` Next.js route handlers (e.g., `src/app/api/(common)/reports/round-fees/pay/route.ts` and `…/revert/route.ts` and the four siblings) does `await prisma.activityLog.createMany(...)` inside a `try/catch`, logs the error, sets `activityLogFailed = true`, and returns success to the user. The activity log is the audit trail that drives the per-row "Activity Log" modal — i.e., the compliance surface for "who paid this row, when, with what remark." A silent failure here means the user is told the row was paid/reverted, no audit row exists, and the modal will show "no history" — a real compliance gap. The correct fix is: (a) emit to outbox inside the same transaction as the summary-service state change so the audit is part of the state transition, or (b) reject the request with 500 when the log write fails (don't lie about success), or (c) write a tombstone "audit write failed" record. Returning `success: true, activityLogFailed: true` is the worst of the three: it propagates the failure to the UI as an extra flag that nothing in the visible diff appears to surface to the user.

**H4. Report rows are loaded with no pagination (the route comment says so explicitly) and no row-count cap.** `src/app/api/(common)/reports/round-fees/route.ts` L8825 comment: "Returns ALL rows for the date range (no pagination)." The same is true of all four sibling routes (CF, IHD, PF, TC). A wide open-ended `from…to` filter on a hospital's daily billing dataset, even with a `gin_trgm_ops` index on `invoice_no`, will pull hundreds to thousands of rows into memory and serialize them through the Next.js handler → summary-api → worker → Prisma pipeline. There is no cursor, no `LIMIT`, no aggregate cap, and no streaming. A misconfigured date range (one year instead of one day) takes down the report page. Add a default `LIMIT 500` and a cursor, or at minimum a server-side cap that returns 413. The same fix applies to the four prior reports — but since this PR also ships the schema, fixing it here also fixes the API contract going forward.

### Medium

**M1. Generic event types are named generically but the worker fan-out coupling is implicit.** The two new event helpers intentionally expose only two event types — `opd_billing.{bill_created, bill_reconciled, refund_created}` and `ipd_daily_bill.{bill_created, bill_reconciled}` — and the doc comment promises "the worker fans a single event out to every registered IPD report." That promise is implemented as a convention, not a contract: there is no registry, no event-type-to-report mapping in this PR, and nothing prevents a future engineer from adding `opd_billing.tax_changed` that only one report cares about (which then silently breaks the others). For five report types that's fine; for fifteen it's a trap. The right time to add the registry is before report #6, not after — one file, ~30 lines, declares which event types each report subscribes to. Optional for this PR; flag for the next one.

**M2. `process.env.DEFAULT_TENANT_ID?.trim()` is read inside every emit call and every `summaryApi` request.** Same pattern in both event helpers and in `summary-api.ts`. This is six reads per HTTP request, four reads per bill mutation. It should be a module-level constant validated once at boot (the `config/index.ts` Zod-validated pattern from `hms-summary-service` already exists). Ponytail: read-once at module load, throw on missing.

**M3. OPD refund emits `opd_billing.refund_created` but the IPD refund path does not.** The new OPD refund emit (`opd-refund.service.ts:7848`) carries `serviceIds` and `procedureIds` to void the right report rows. The diff also adds IPD event emits across cathlab, ward, daily-bill, service-request, and proxy-bill — but no IPD refund path was touched. If IPD refunds exist (the four prior fee reports all expose `source_voided_reason ∈ {CANCELLED, REFUNDED, REMOVED_ON_EDIT}` so they expect refund events), then a refund today silently keeps report rows alive with a non-voided status. Either confirm IPD refunds route through one of the touched paths and the worker deduces the void from the daily-bill reconcile, or add the symmetric IPD refund emit in a follow-up.

**M4. The five UNIQUE-per-grain-column indexes are not enforced by Prisma and can drift from the schema.** Per the migration comment ("hand-added: the Prisma schema language has no syntax for CHECK constraints, partial indexes, or pg_trgm GIN indexes"), these constraints are migration-only and won't appear in a fresh `prisma db pull`. The `summary-service` schema is a hand-maintained subset of HMS; if anyone regenerates the subset from a `prisma db pull`, these indexes disappear from the next migration but the table-level UNIQUE constraints stay — fine. But the inverse direction (pulling into the summary-service's local schema, then someone re-running a fresh migration on a different env) is a foot-gun. Add a CI check that asserts the indexes exist after a `prisma migrate deploy`, or at minimum a `// do not regenerate` marker in `schema.prisma` near the relevant models.

### Low / Nit

**L1.** The two event-emit helpers and the summary-api helper all duplicate the same `getTenantId()`-style read pattern (`opd-billing-events.ts:9208`, `ipd-daily-bill-events.ts:9149`, `summary-api.ts:9242`). One module-level constant solves M2 and removes the duplication. Ponytail: `const TENANT_ID = process.env.DEFAULT_TENANT_ID ?? throwOnBoot()`.

**N1.** `enhancedApiHandler` is invoked five times per report × five reports = 25 times in this PR, with identical auth+validation scaffolding except the permission subject. The wrapper accepts a single `permissions: [...]` literal per call site, so a permission matrix table would have been shorter. Optional.

**N2.** `safeJson` in `summary-api.ts` swallows JSON parse errors and returns `null`; if the summary-api ever returns malformed JSON on a 2xx (e.g., a bug), the caller gets a TypeError cast downstream rather than a clear error. Throw or log.

**N3.** The `summary-api.ts` comment block (lines 9236-9239) explains *why* HMAC is removed but lives in the wrong place: it should be a `CONTEXT.md` / `docs/adr/` ADR, not a code comment. Code comments rot; ADRs don't.

## Recommendation
1. **Re-add HMAC** between HMS and summary-api (C1) before merging — this is the kind of regression that gets a CVE-issued advisory six months later.
2. **Fix the `DEFAULT_TENANT_ID` soft-fail** (H1): throw at boot or throw inside the transaction. "Log and continue" is not a control.
3. **Prove the UNIQUE-on-grain-columns pattern** (H2) — run the worker against a representative day of production data with each report type and confirm no `ON CONFLICT` collision.
4. **Make audit failures loud** (H3) — return 5xx, not `success: true, activityLogFailed: true`.
5. **Add a server-side row cap** (H4) — even 500 rows is a sane default until cursor pagination lands.
6. Consider the event-subscription registry (M1) before the next fee-report PR; the M2/L1 deduplication and N3 ADR move are good follow-up cleanup PRs.

The structural pattern is sound and the rest of the series should fold into it cleanly — this PR mostly works, but the four Highs plus the Critical are not a "land and iterate" set; they're a "block and fix" set.
