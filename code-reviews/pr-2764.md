## Summary

When a service was created with `serviceType = LAB_TEST`, a row was inserted into `lab_tests` in the same transaction. The legacy `deleteService` repository method called `prisma.service.delete` and translated the resulting FK violation into a friendly 409 — but for *newly created* Lab Test services there is no `lab_groups` row, so the chain `service → lab_tests → (nothing)` was fine. The bug, on closer reading, is that the old `deleteService` did `prisma.service.delete` *first*; Postgres then has to cascade the FK from `lab_tests` back to `service`. Even with no `lab_groups` history, some prior insert path apparently left state that caused the cascade to fail (likely the `lab_template_items.labTestId` FK was non-null in some migration window, or the cascade direction was the reverse of what the code assumed). The PR replaces the failing path with a new `deleteServiceWithDependencies` that:

1. Looks up the `labTest` for this service inside a transaction,
2. Counts referencing `labGroup` rows (i.e. `lab_groups.labTestId`),
3. Throws the same friendly 409 if any exist (preserves the old UX), and
4. Otherwise deletes the `labTest` row, then the `service` row.

The new method is invoked from `service.service.ts:212`, replacing the single call site. The old `deleteService` is left in the repository but has no remaining callers.

## Strengths

- **Root-cause fix is in the right place.** The new method lives in the repository (not the service), so it stays close to the SQL semantics and to the `prisma.$transaction` boundary. The `service.service.ts` change is a one-line swap.
- **Idempotent / safe shape.** `findFirst` then conditional `delete` is straightforward and avoids the "blindly delete then fail" pattern of an earlier revision of this same PR (the 2026-06-19 review flagged that the previous version of `deleteServiceWithDependencies` did `deleteMany` first and *lost* the friendly 409 for services with history — that regression is now fixed).
- **Preserves the existing 409 message.** `"This service has previous history. You cannot delete right now."` matches the old `deleteService` exactly, so front-end error handling (and the original ticket) keeps working.
- **Transaction scope is tight.** Both the count and the deletes happen inside one `prisma.$transaction`, so a concurrent `labGroup` insert between the count and the `labTest.delete` would either be serialized by `FOR UPDATE` (Prisma's default) or produce a P2003 on commit. Either way no orphan rows.
- **No new types, no `any`, no schema change.** Diff is constrained to the two files advertised in the PR title.
- **No new `authorizeProcedure` bypass introduced.** The single call site is `service.service.ts:212`, which is already invoked from an authorized tRPC procedure (verified via the file's other read paths); authorization isn't this PR's concern.

## Root cause analysis (the bug, not the fix)

The PR title says "newly created Lab Test services" can't be deleted. The likely sequence:

1. `createService` writes a `service` row and (in the `LAB_TEST` branch) a `lab_tests` row in the same transaction.
2. A user clicks Delete on this service. The front end calls `deleteService` (the service-layer method).
3. The old `deleteService` did `prisma.service.delete({ where: { id } })`. With `lab_groups.labTestId` `ON DELETE RESTRICT` and no `lab_groups` row yet, this *should* have succeeded — the FK goes the wrong way for a RESTRICT to trigger. So either: (a) the real culprit is `lab_template_items.labTestId` if that FK is `ON DELETE RESTRICT` (worth checking the latest migration; the 2026-03-12 migration only created the `lab_groups` FK), or (b) some other table — possibly `lab_service_item` (which has `serviceItemId → service` via the `Service` join), or `speical_lab_tests` — was created in a different migration and has an FK that fires on the `service` delete.
4. The new path's explicit "find `labTest`, then delete it, then delete `service`" sequence sidesteps whatever the missing FK was by explicitly removing the dependent row in dependency order.

Whatever the precise original cause, the new code is correct: it now always deletes the `labTest` row *before* the `Service` row when no history exists, and *does not* delete it (and throws 409) when history exists. That's the right shape.

## Side effects / what could be protected that shouldn't be deleted

- **`lab_template_items.labTestId` is `ON DELETE SET NULL`** in the current schema (per the existing 2026-06-19 review of this same PR), so deleting a `labTest` row will null out the FK on template items but preserve the template item itself. That matches the old behaviour because the old code never reached this point — verify with the team that "delete Lab Test service → wipe the `labTest` row → null out any template references" is the intended product behaviour and not just a side effect.
- **No audit row in `lab_tests` history** — the `labTest.delete` is hard. If the team wants soft-delete or history preservation for audit (the activity log entry helps, but the row itself is gone), flag for product.
