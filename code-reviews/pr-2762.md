## H1. `updateWardServicePrices` / `updateWardProcedurePrices` no longer persist `IPDDailyBill.totalAmount` / `subtotal`

**Where:** `src/app/(dashboard)/shared/ipd/services/ipd-daily-bill.service.ts:906-988` (post-diff)

Old path:

```ts
await this.wardServiceRepository.updateServiceItems(servicePriceUpdates, transactionClient);
await this.ipdDailyBillRepository.updateIPDDailyBill(dailyBillId, { updatedById: userId }, transactionClient);
await this.recalculateDailyBillTotalAndAdjustDeposit(dailyBillId, userId, transactionClient);
// -> inside: calculateTotalAmount(dailyBill), persist totalAmount + subtotal,
//    then adjust deposit by (oldTotal - newTotal)
```

New path (and the symmetric `updateWardProcedurePrices`):

```ts
const itemIds = servicePriceUpdates.map((s) => s.id);
const oldItems = await this.wardServiceRepository.findServiceItemsByIds(itemIds, transactionClient);
let itemTotalDifference = 0;
for (const update of servicePriceUpdates) {
  const oldItem = oldItems.find((item) => item.id === update.id);
  if (oldItem && update.amount !== undefined) {
    itemTotalDifference += update.amount - oldItem.amount;
  }
}
await this.wardServiceRepository.updateServiceItems(servicePriceUpdates, transactionClient);
await this.adjustDepositByLineItemDifference(dailyBillId, itemTotalDifference, transactionClient);
// adjustDepositByLineItemDifference: fetches daily bill, refunds/deducts deposit.
// It does NOT call ipdDailyBillRepository.updateIPDDailyBill for totalAmount / subtotal.
```

The new `adjustDepositByLineItemDifference` only adjusts the deposit — it does not recompute `calculateTotalAmount(dailyBill)` and does not persist `totalAmount` / `subtotal`. The two `updateIPDDailyBill({ updatedById })` calls that used to live on these paths were also removed.

The persisted `IPDDailyBill.totalAmount` / `subtotal` columns now go stale after every ward-service price edit. List views that bypass `getDailyBillList` (e.g. `daily-bill-list-table.tsx:139` reads `dailyBill.totalAmount` from the row directly) will show the previous total. More importantly, anything downstream that uses the stored `totalAmount`/`subtotal` for invoicing, discharge summaries, or reconciliation will be off by the line-item delta while the deposit has already moved by exactly that delta — a guaranteed drift between the bill total and the deposit ledger.

**Fix:** keep the daily-bill total recalculation. Easiest path: in `adjustDepositByLineItemDifference`, after the deposit is updated, also recompute and persist `totalAmount` / `subtotal`. At minimum:

```ts
await this.ipdDailyBillRepository.updateIPDDailyBill(dailyBillId, {
  totalAmount: calculateTotalAmount(dailyBill),
  subtotal: calculateTotalAmount(dailyBill),
  updatedById: userId,
}, transactionClient);
```

before returning, so the stored columns stay in sync with what `calculateTotalAmount(dailyBill)` would return.

## H2. `recalculateDailyBillTotalAndAdjustDeposit` passes the outer `tx` to util methods instead of the inner `transactionClient`

**Where:** `src/app/(dashboard)/shared/ipd/services/ipd-daily-bill.service.ts:738-779` (post-diff)

```ts
async recalculateDailyBillTotalAndAdjustDeposit(
  dailyBillId, userId,
  tx?: Prisma.TransactionClient,
) {
  await this.runInTransaction(tx, async (transactionClient) => {
    // ... updates daily bill via transactionClient ...
    if (difference > 0) {
      await this.utilService.addedReturnedAmountToDeposit(
        dailyBill.admissionId, difference,
        { tx, /* transactionMetaData */ },        // <-- `tx` (outer param)
      );
    } else if (difference < 0) {
      await this.utilService.calculateDepositAmount(
        dailyBill.admissionId, Math.abs(difference),
        { tx, /* transactionReason, transactionMetaData */ },  // <-- `tx` (outer param)
      );
    }
  });
}
```

The `runInTransaction` helper creates `transactionClient` for the work in this method. But `addedReturnedAmountToDeposit` / `calculateDepositAmount` are called with `{ tx }` where `tx` is the **outer** parameter — undefined when the method is invoked standalone. When that happens, the util methods call `findLastestDeposit` / `updateDepositAmount` / `createDepositTransactionHistory` **outside** the transaction that just updated `IPDDailyBill.totalAmount`. The bill update and the deposit adjustment can no longer be rolled back together.

Every internal caller happens to pass the active `transactionClient`, so the bug is latent right now, but the method is public and is also a footgun for any future caller. Compare with the new sibling `adjustDepositByLineItemDifference` (also in the diff) which correctly uses `tx: transactionClient` inside its own `runInTransaction` block.

**Fix:** pass `transactionClient`, not `tx`:

```ts
{ tx: transactionClient, transactionMetaData: { ... } }
```

(and likewise in the `difference < 0` branch).

## H3. `isPriceFromMapping` for ward service procedures is now derived as `!doctorId`, making doctor-bound procedures editable in the daily bill

**Where:** `src/app/(dashboard)/shared/ipd/services/ward-service.service.ts:412-421` and `:767-774` (post-diff)

```diff
   proceduresToCreate.push({
     ipdWardServiceId: id,
     procedureId: newItem.procedureId,
     doctorId: newItem.doctorId ?? null,
     qty: newItem.qty ?? 1,
     price: newItem.price ?? 0,
-    isPriceFromMapping: newItem.isPriceFromMapping ?? (newItem.price ?? 0) > 0,
+    isPriceFromMapping: !newItem.doctorId ? true : false,
     ...
   });
```

`isPriceFromMapping` controls editability in the daily bill view (`daily-bill-ward-services-and-procedures.tsx`: `service.isPriceFromMapping !== false` toggles the `<NumberInput>` on/off, and the outer "edit" button is gated by `hasEditableManualPriceServices || hasEditableManualPriceProcedures`).

Old: `isPriceFromMapping ?? (price > 0)` — if the form supplied the flag, honour it; otherwise fall back to "price is non-zero".

New: `!doctorId ? true : false` — every procedure that has a doctor assigned is now `isPriceFromMapping = false`, regardless of whether its price actually came from a doctor mapping, a manual override, or was already set to zero.

Net effect: a procedure mapped to a doctor (where the price should be considered "from mapping") now exposes its price to the daily-bill editor, so the user can type a custom number that overwrites the doctor mapping price in `IPDWardServiceProcedure`. The doctor's mapping row is never updated — the ward-service row and the doctor-mapping row will disagree on price, and the next time the form rebinds from DB, the user will see a different value than what they typed. There's no signal in the data model that this happened.

The service-side change (line 340-341) drops the `??` fallback entirely and forces `isPriceFromMapping: (price > 0)` for ward service items, which has the inverse problem: any manually-overridden service with a non-zero price is now considered "from mapping" and becomes uneditable in the daily bill.

**Fix:** for the procedure branch, restore the original semantics and stop overloading the flag:

```ts
isPriceFromMapping: newItem.isPriceFromMapping ?? !newItem.doctorId,
```

(or, if the intent is to track whether the procedure was bound to a doctor at create time, rename the column — the current code can't distinguish "doctor-bound" from "manually entered but no doctor").

For the service branch (line 340-341), keep `newItem.isPriceFromMapping ?? (price > 0)` — that's what the field was designed for.
