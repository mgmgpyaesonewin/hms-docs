## Summary

This PR closes all four findings from the 2026-06-17 cathlab code review: the duplicated `updateCathLabServiceItems` / `updateCathLabProcedureItems` bodies are collapsed behind a new generic `CathLabItemService.updateItems`, the action that previously validated against the wrong schema now uses its own `updateCathLabProcedureItemSchema`, the dead/imported-but-unused schema file is filled in with a real definition, and the optimistic-state commit in `handleSaveAllChanges` is moved out of the synchronous handler so it only fires after the server confirms (`useAction`'s `onSuccess`).

On top of those fixes the PR adds the actual feature the ticket asks for: services (`CONSUMABLE_1`) and procedures in the daily-bill cathlab tab are now editable. Procedures get a new `<NumberInput>` column gated by `item.doctor && (item.price === 0 || editingProcedureIds.has(item.id))`, services get the same gate minus the doctor check. The two new server actions are wired through `authActionClient` and routed into the helper. The diff also threads `doctor: { id, title, user.fullName }` through `map-daily-bills.ts`, `reshapeCathLab`, and the discharge / final-bill / daily-bill detail validators, fixes a long-standing bug where `proc.id` in the reshaped daily bill was set to `proc.procedure.id` (the *procedure catalog* id, not the *procedure-item row* id), and relaxes the company-direct price cap from 100 → no upper bound (line 654).

The headline finding is that the extraction is **partial**, not full. `updateCathLabCompanyDirectItems` is *not* migrated to the new helper; it still has its own 100+ line inline transaction, still calls `this.ipdDailyBillRepository.findDailyBillById` directly, and still computes the deposit delta with a custom commission formula that no other code path uses. The diff explicitly removes a commented-out earlier draft of the function but leaves the live one untouched. This means the next contributor will either (a) duplicate the deposit-update logic a third time, or (b) extract it in a third PR. Either is fine, but the helper's `case "company-direct"` arm in `fetchExistingItems` is now dead code that pretends to support a path the service never invokes. Plus the `ItemUpdateConfig.requireDoctorAndZeroPrice` flag is set by both callers but never read inside the helper — the doctor-required check is implemented as an inline `validateFn` callback that only checks `existing.doctorId`, **not** `existing.doctorId && (incoming price === 0 || existing price === 0)`. The PR body says the rule is "doctor + price=0"; the implementation is "doctor exists", which is materially weaker.

Other notable concerns: (1) every caller now does its own `findMany` *and* the helper does another `findMany` for the same `id: { in: [...] }` — two queries per save, both inside the same `$transaction`; (2) the procedure items' editability gate at `daily-bill-cathlab.tsx:546` reads `item?.doctor` but the procedures type in `DailyBillDetailData` (line 312) makes `doctor` **non-nullable** for newly-selected procedures, while the existing `cathLab.service.ts:1359-1370` (`effectivePrice = p.doctorId ? 0 : (p.price ?? 0)`) stores `doctorId` as nullable — so the runtime shape disagrees with the type; (3) the `handleSaveAllChanges` UI still fires `setOriginalServices(editableServices)` *twice* in the same `onSuccess` (lines 112-115 of the diff, both `setOriginalServices(editableServices)` are present) — the second call is harmless but it's clearly a paste mistake; (4) no new tests for the helper, no tests for the procedure-item action, and the existing `daily-bill.helper.test.ts` (if any) is not extended for the new `hospitalPercentage`/`hospitalAmount` branch.

The shape is right and the data flow works for the happy path — a manager edits a procedure's price from 0 → 1500, the schema accepts, the helper updates, the deposit is recalculated, the audit row is written. But the partial extraction and the un-checked `requireDoctorAndZeroPrice` flag will let a procedure with `doctorId === null` slip through the "doctor required" guard if any future caller forgets to pass the `validateFn` (which is exactly what the generic helper is designed to enable), and the bug surface that opens up — pricing a procedure without a doctor for an IPD patient — is one of the financially-sensitive flows the cathlab review from a week ago was trying to tighten.

- **`src/app/(dashboard)/shared/cathlab/services/cathlab.service.ts:1193-1235, 1518-1550` — Each service path runs `findMany` twice per save (once in the caller, once inside the helper)**
  `updateCathLabServiceItems` opens with `const existingItems = await tx.cathLabServiceItem.findMany({ where: { id: { in: itemIds } }, include: { cathLab: { select: { id: true } } } });` (`cathlab.service.ts:1197-1205`). Then it passes `payload.items` into `cathLabItemService.updateItems(...)`, which immediately calls `fetchExistingItems(...)` (`cathlab-item.service.ts:768-772`) which runs the **same** `tx.cathLabServiceItem.findMany({ where: { id: { in: ids } }, include: { cathLab: { ... } } })` again. Same for procedures (`cathlab.service.ts:1519-1527` → `cathlab-item.service.ts:768-772`).

  Fix: remove the caller's pre-fetch (`cathlab.service.ts:1197-1205` and `:1519-1527`); change `updateItems` to return `cathLabId` in the result, then use it for the audit. Or extract `cathLabId` from the first existing item the helper returns and write the audit before `updateItems` returns. Either way, one query per save.
  Evidence: `cathlab.service.ts:1197-1205` — first `findMany`; `cathlab-item.service.ts:857-888` — `fetchExistingItems` runs the same `findMany` again. Same pattern duplicated for procedures.

- **`src/app/(dashboard)/shared/ipd/helpers/daily-bill.helper.ts:31-39` — `hospitalPercentage` / `hospitalAmount` adds-on-discount ordering silently disagrees with the inline cathlab.service.ts:783-790 calculation**
  The helper now applies hospital markup **before** discount:
  ```ts
  let amount = price * payableQty;
  if (item.hospitalPercentage != null && item.hospitalPercentage > 0) {
    amount += (amount * item.hospitalPercentage) / 100;
  } else if (item.hospitalAmount != null && item.hospitalAmount > 0) {
    amount += item.hospitalAmount;
  }
  if (item.discountPercentage != null && item.discountPercentage > 0) {
    amount -= (amount * item.discountPercentage) / 100;
  } else if (item.discountAmount != null && item.discountAmount > 0) {
    amount -= item.discountAmount;
  }
  ```
  But the inline company-direct transaction in `cathlab.service.ts:761-790` (called from `updateCathLabCompanyDirectItems`, *not* from `calculateBillLineItemAmount`) computes commission as `commission = (newBaseAmount * hospitalPercentage) / 100` and then `let newFinalAmount = oldFinalAmount + commission`. That's a **different** formula: commission is computed off the *base amount* (price × qty, no foc adjustment), but the helper computes it off `payableQty` (price × max(0, qty - foc)). For an item with `foc > 0`, the helper's number is **lower** than the inline formula's number. The two paths disagree on the final billed amount for the same input.
  This matters because the inline path (company-direct) is what actually writes to `cathLabCompanyDirectItem.amount` today (and is what the daily bill detail then reads). The helper path is a "read-side" recompute that runs against `BillLineItemInput` in `calculateBillLineItemAmount` somewhere — likely in the discharge / final-bill aggregation. If the discharge aggregator uses `calculateBillLineItemAmount` and the daily-bill-detail UI uses `existing.amount` (which was written by the inline path), the displayed total on the discharge screen will not match the persisted total on the daily bill. That's a real money discrepancy.
  Fix: pick one formula. Document the order of operations (markup first, then discount, with foc applied to base). Apply it consistently in both places. Likely the helper is wrong: hospital commission should be on the *post-foc base*, but the `+= amount * pct` order means it's also affected by the discount in the discount branch, which is also weird (a 10% discount on a 10%-markup-up item reduces both the discount base and the markup-corrected amount).
  Evidence: `daily-bill.helper.ts:31-39` — helper path; `cathlab.service.ts:761-790` — inline path. Both compute "amount" for the same `BillLineItemInput` shape and disagree on foc handling and discount-after-markup.

  The schema `updateCathLabProcedureItemSchema: z.object({ ..., price: z.number().min(0).optional(), ... })` accepts `undefined` (field omitted) or a `number >= 0`. `null` is **not** accepted — `z.number()` rejects `null` before the `.min(0)` runs.   Evidence: `daily-bill-cathlab.tsx:541-546` — gate; `update-cathlab-procedure-item.schema.ts:7` — `price: z.number().min(0).optional()`.

- **`src/app/(dashboard)/ipd/daily-bill-list/[id]/features/components/daily-bill-cathlab.tsx:181-187, 227-228` — `handleServicePriceChange` and `handleProcedurePriceChange` set `Math.round(finalAmount)` for the UI display but the helper will recompute it server-side**
  The UI rounds `finalAmount` for `amount` and `netAmount` (services only — procedures only set `amount`). The server-side helper then recomputes `amount = Math.max(0, calculatedAmount)` (`cathlab.service.ts:1412` and `:1677`). If the server's recomputation produces a different value (e.g. because `payableQty` differs from what the UI used, or because the discount branch differs), the UI's optimistic value will be overwritten on `onSuccess` via `queryClient.invalidateQueries`. Today they agree, but the rounding in the UI is doing double duty — it's both the optimistic value *and* what's sent in the payload (`amount: item.amount`). The UI is sending the rounded value; the server is re-deriving it; if those diverge by a few kyats due to rounding (the server doesn't always round), the user sees a flash of the rounded value followed by the server's value. Pick one — either (a) round in the server only (drop the UI rounding), or (b) round in both but make sure the rounding logic is identical (use `Math.round` everywhere, including the discount step).
  Evidence: `daily-bill-cathlab.tsx:206-207` — UI rounds for service items; `:232` — UI rounds for procedure items; `cathlab.service.ts:1412` — server rounds with `Math.max(0, calculatedAmount)` (no explicit `Math.round`).

The procedure path checks `if (itemIds.length === 0)` and returns the early-success message. The real issue: the procedure path's "empty items" guard at lines 1534-1544 returns a different shape than the helper's normal result (it returns `items: []` instead of `items: result.items`, but `result.items` is the array of updated rows that the caller never reads). This is harmless — both shapes are passed through. But the duplication of `if (itemIds.length === 0)` between the two callers (service-items doesn't have it, only procedures) is asymmetric and worth deleting from procedures for consistency.
  Evidence: `cathlab.service.ts:1534-1544` — procedure early-return; `cathlab.service.ts:1203-1205` — service items has no equivalent guard.

- **`src/app/(dashboard)/shared/cathlab/services/cathlab.service.ts:751` — Pre-existing `this.logger.info("PAYLOAD", payload.items);` is now exercised on every save but the structured logging wrapper is already in place**
