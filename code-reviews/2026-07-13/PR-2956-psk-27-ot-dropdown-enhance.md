## Summary
Replaces bulk-loaded `Select` dropdowns (which fetch every record upfront) with new `*SearchSelect` components backed by an infinite-scroll search hook for the OT / Cathlab request forms. Adds a generic `<SearchSelect<T,D,F>>` component that uses `useInfiniteQuery` with debounced search, `keepSelectedItem` to preserve selection across re-renders, and a `fetchItem(id)` fallback that hydrates a not-in-page selected value via a per-entity by-id endpoint. The PR also adds two new REST endpoints (anesthesia-type and user by id), an anesthesia-type service/repo read path, and a `useUserSearchInfinite` hook that fuzzily ranks users via `sortEntitiesBySearch`. Tests cover module-department filtering, schema parsing, and a "no longer uses these symbols" source-grep assertion on the new components.

## Issues

### Critical
- **`assistantDoctors` field reports the wrong validation error** — at `src/app/(dashboard)/ot/request-list/features/components/ot-request-form.tsx` ~line 1133 (and duplicated at `src/app/(dashboard)/cathlab/request-list/features/components/cathlab-request-form.tsx` ~line 1130 and `src/app/(dashboard)/ipd/features/components/service-request/cathlab/cathlab-request-form.tsx` ~line 1130), the new `DoctorSearchSelect` renders `error={form.formState.errors.cardiologists?.[index]?.cardiologistId?.message}` for the **assistant doctor** input. The error path points to the cardiologists array, so an assistant-doctor validation error will silently never surface. The pre-refactor code correctly read `assistantDoctors?.[index]?.assistantDoctorId?.message`. This was a copy-paste during the migration and is duplicated in 3 files.

### High
- **Massive commented-out dead code left in committed source** — `cathlab-request-form.tsx` keeps the entire 235-line pre-refactor block (the old `AssistantNurseSelect` component, all `useMemo`/`useState`/`useDebouncedCallback`/`makeFetchClinicsQuery`/`makeGetUsersQuery`/`makeFetchAnesthesiaTypesQuery` plumbing, and the old `Select` JSX blocks) as `//` comments. The diff also comments out (rather than deletes) 18 imports across the changed files. This is the bulk of the +1303 additions. Either delete the block or split it into a separate commit/PR; do not land a PR where ~50% of the "code" is commented-out ghosts.
- **`SearchSelect.fetchItem` is not Promise-or-T** — the `fetchItem` prop is typed `(id: string) => Promise<T> | T` but every concrete caller (`patient`, `doctor`, `clinic`, `user`, `anesthesia-type`) returns a `Promise`. The sync branch and the `if (result instanceof Promise)` branching are dead complexity. Tighten to `(id: string) => Promise<T>` (or accept the existing `Promise<T> | undefined` shape from `apiClient`).
- **`SearchSelect.fetchedIds` race allows duplicate in-flight fetches** — `search-select.tsx`: the `finally` block deletes the id from `fetchedIds.current` after success/error, so if the value changes twice in quick succession to the same id (or the effect runs twice with the same id, e.g. when `allItems` updates after a successful search), the second invocation will fire its own request while the first is still in flight. Net effect: one extra `/api/<resource>/:id` round-trip per dropdown with a preselected-but-not-in-page value, on detail pages this multiplies per select. Use a `Set` of currently-in-flight ids and a separate "resolved" cache, or just guard with a fetch-status ref keyed by id.

### Medium
- **Same diff applied to two `cathlab-request-form.tsx` files** — `src/app/(dashboard)/cathlab/request-list/features/components/cathlab-request-form.tsx` and `src/app/(dashboard)/ipd/features/components/service-request/cathlab/cathlab-request-form.tsx` receive identical edits. The wrong-error-path bug above is duplicated in both. Either collapse into a single shared component or accept the duplication but acknowledge it in the PR description; do not silently copy-paste a form twice.
- **`users-repository.ts` mints results inside the slice** — `take: limit === 0 ? undefined : limit * 10` plus `sortEntitiesBySearch(...)` plus `slice(0, limit)` silently changes the public `getUsersList` contract: callers that pass `limit > 0` now get the top-N search-ranked subset of a 10x window, not the first `limit` ordered-by-`createdAt desc` rows. Every downstream selector (the user-search dropdown, nurse dropdowns, etc.) will see different data ordering. The previous behavior was explicit "newest first"; this one is "best match first within a 10x window". Document it in the schema/CHANGELOG, or scope the change to the user-search-select flow rather than touching the shared repository's read path.
- **`usePatientSearchInfiniteForOT` lives in `shared/proxy-bill/hooks/`** — `ot-form.tsx` imports `usePatientSearchInfiniteForOT` from `@/app/(dashboard)/shared/proxy-bill/hooks/use-patient-search-for-module` because it gates `patientType = OT`. The hook name and "for OT" filtering is OT-specific; housing it under proxy-bill makes every OT consumer depend on a proxy-bill feature module. Move the hook to `src/app/(dashboard)/shared/patient/hooks/` (or similar) with a `module` filter param, not a hardcoded OT-name.
- **Test references a file not modified in this PR** — `src/app/(dashboard)/shared/ot/__tests__/ot-dropdown-infinite-components.node.test.ts` iterates over `src/app/(dashboard)/opd/opd-billing/ot-request-list/features/components/opd-ot-request-form.tsx`, but that file is not in this PR's changed-files list (33 files changed). Either the file was meant to be updated and got missed (a real bug), or the test was meant to be removed/skipped. Right now the test asserts against a file that wasn't changed to use the new components, so the assertion either fails or is silently a no-op.
- **`SearchSelect` `iconOptions.position` default flipped from `"left"` to `"right"`** — silent behavior change for any caller that omits `iconOptions` (e.g., the new `ClinicSearchSelect` and `UserSearchSelect` show their icon on the right by default now). Verify visually on every page that previously had `searchable leftSection={<Search />}` — the icon moved.

### Low / Nit
- **Source-grep test is brittle** — `ot-dropdown-infinite-components.node.test.ts` reads the source text and asserts presence/absence of `usePatientSearchInfiniteForOT`, `makeFetchPatientsQuery`, etc. This couples the test to character-for-character source content; a rename or formatting change unrelated to the migration will break it. Replace with a behavior assertion (render the component, mock the hook, assert the right hook was selected) or delete.
- **Magic string `"__loader__"` sentinel in `MainProcedureSearchMultiSelect`** — the multiselect pushes a synthetic "Loading..." option with `value: "__loader__"` to drive a custom `renderOption`. Cleaner: hoist loading into the multiselect's empty/listbox footer using Mantine's render hooks; or use `limit` based pagination state. The current shape leaks a sentinel value to whoever reads `value` from `onChange` (the caller filters it out, but it's an invisible contract).
- **`MainProcedureSearchMultiSelect` shows `selectedValue` (UUID) as label for missing items** — `if (!mapped.some(...)) { mapped.push({ label: selectedValue, value: selectedValue }); }` displays a raw id when a previously-selected procedure isn't in the current search page. Users see `ab12cd34-...` in the chip. Use `fetchItem` (the same pattern SearchSelect uses) to hydrate, or surface a "Loading previous selection" placeholder.
- **`PatientSearchSelect.getItems` wraps a flatMap in a dead arrow body** — `getItems={(data) => { const patient = data?.pages.flatMap(...) ?? []; return patient; }}` — the `patient` const adds nothing; collapse to `getItems={(data) => data?.pages.flatMap((p) => p.result?.patients ?? []) ?? []}` (or the previously-cleaner one-liner that this PR replaced).
- **Inconsistent duplicate-item toast wording** — `ipd-emr-pharmacy-request-form.tsx` toast says `"Pharmacy Item already added."` and `ipd-emr-prescription-form.tsx` says `"Prescription Item already added!"` — both new. Pick one voice.
- **`AnesthesiaTypeSearchSelect` typing dances a `infer D` chain on a hook return** — `type AnesthesiaTypeSearchData = ReturnType<typeof useAnesthesiaTypeSearchInfinite> extends { data: infer D } ? D : never` is a 5-line indirection to pull a `data` shape that's already exported. Export the data type from `use-search-select-infinite.tsx` (or from `useEntitySearchInfinite`) and use it directly; remove the infer dance from every per-entity select.
- **`MainProcedureSearchMultiSelect` skips `error`/`status` propagation** — other `SearchSelect` users rely on `value ?? null`; this multi-select sets `value={value}` directly on the inner `MultiSelect` and lets Mantine warn about null. Pass `value={value ?? []}` for consistency.
- **No `patientTypesFilter` consumer outside `PatientSearchSelect`** — the prop is added but no caller in this PR passes it. Either delete until needed (`yagni`) or wire one call site so the contract is tested end-to-end.

3. Tighten `fetchItem` typing to `Promise<T>` and resolve the in-flight fetch race using a single `Map<id, Promise<T>>` (or equivalent) — current implementation can fire duplicates.
4. Either update `opd-ot-request-form.tsx` to match the test or remove that assertion from `ot-dropdown-infinite-components.node.test.ts`. Right now the test asserts against unchanged code.
5. Decide where `usePatientSearchInfiniteForOT` belongs. It's OT-shaped but lives under proxy-bill/hooks. Move to a neutral location or rename to something module-agnostic with a `module` param.
6. Move `getUsersList`'s new search-rank + 10x take semantics behind a new method or feature-flag it; surface the behavior change in the ticket so consumers know "first-N" is now "top-N within 10x window".
7. Verify visually that `<Select searchable leftSection={<Search />}>` consumers (HD, OPD prescription, etc.) still look right after the `iconOptions.position` default flip from `left` → `right`.
