## H1. Same `dischargeTime` formatter bug still present in two sibling detail pages

**Where:**
- `src/app/(dashboard)/cathlab/request-list/[id]/page.tsx:169`
- `src/app/(dashboard)/opd/opd-billing/cathlab-request-list/[id]/page.tsx:197`

The PR fixes one occurrence of `formatToDefaultDate(cathLabRequestData.dischargeTime!)` in `ipd-cathlab-detail.tsx:186` (swapped to `formatToDefaultTime`). Two identical copies of the bug remain in the cathlab and OPD detail pages:

```tsx
<Text fw={700}>Operation Time: </Text>
{formatToDefaultTime(cathLabRequestData.operationTime!)}
</div>
<div className="flex gap-2">
  <Text fw={700}>Discharge Date: </Text>
  {formatToDefaultDate(cathLabRequestData.dischargeDate!)}
</div>
<div className="flex gap-2">
  <Text fw={700}>Discharge Time: </Text>
  {formatToDefaultDate(cathLabRequestData.dischargeTime!)}   // bug
</div>
```

`dischargeTime` is a full `DateTime` (built in `cathlab-request.repository.ts:366-369` by `combineDateWithTime(payload.dischargeDate, payload.dischargeTime)`), not a string like `"10:30"`. Rendering it with `formatToDefaultDate` (`DD/MM/YYYY`) drops the time-of-day and prints the same date as the `Discharge Date` line directly above it — exactly what the IPD fix addresses. Users on the cathlab request-list detail and the OPD billing cathlab detail pages still see a duplicated date where the discharge time should be.

**Fix:** swap to `formatToDefaultTime` on both lines (same one-character change as the IPD file).

```tsx
// cathlab/request-list/[id]/page.tsx:169
{formatToDefaultTime(cathLabRequestData.dischargeTime!)}

// opd/opd-billing/cathlab-request-list/[id]/page.tsx:197
{formatToDefaultTime(cathLabRequestData.dischargeTime!)}
```

## H2. Sale-return mocks for `approvedBy` / `rejectedBy` no longer match the rendered field after the `fullName → username` swap

**Where:**
- `src/app/(dashboard)/pharmacy/sale-return/__mocks__/sale-returns-handlers.mock.ts:50-57, 129-136`

The PR moves the rendered field on the Status select from `approvedBy.fullName` / `rejectedBy.fullName` to `approvedBy.username` / `rejectedBy.username`, and widens the `Pick<User, ...>` type and the Prisma `select` to include `username`. The MSW mock handlers still only set `fullName` and `id`:

```ts
approvedBy: {
  fullName: "Approver User",
  id: "user3",
},
rejectedBy: {
  fullName: "Rejector User",
  id: "user4",
},
```

The MSW handler is consumed by UI tests (`http.get(...)` in `sale-returns-handlers.mock.ts` powers the components under `pharmacy/sale-return/features/**` that render the Status column). After this PR, those UI tests will see `undefined` rendered in the APPROVED/REJECTED badge's user label — a silent regression in test coverage of the Status cell. `username` is a non-nullable field in `prisma/schema.prisma` (`String @unique @map("username")`), so adding it to the mocks is safe and matches what the repository now returns.

The `sale-returns-service.node.test.ts` fixture is a raw Prisma input, not the `SaleReturn` UI type, so the missing field is harmless there. Only the MSW mocks need updating.

**Fix:** add `username` to both `approvedBy` and `rejectedBy` in `sale-returns-handlers.mock.ts` (both occurrences at lines 50-57 and 129-136):

```ts
approvedBy: {
  fullName: "Approver User",
  username: "approver_user",
  id: "user3",
},
rejectedBy: {
  fullName: "Rejector User",
  username: "rejector_user",
  id: "user4",
},
```
