# PR #2784 — feat: add pharmacy request functionality for OPD and ED (direct + prescription)

**Author:** DaDDy-chilll (Paing Sett Kyaw)  **Branch:** → main  **Files changed:** 18 (source: ~860 LOC net; package-lock churn: +1271/-2254)

## Summary

Adds a "direct" pharmacy-request entry point (no prescription required) for OPD and ED tabs alongside the existing prescription-driven flow. New `requestType=direct|prescription` query param routes the shared `EmrPharmacyRequestForm` into a free-form item picker or the legacy prescription-seeded view. Also hardens `EmrPharmacyRequestTransferService.cancel` so a request can only be cancelled while the pharmacy transfer is still `PENDING`, and adds two new pages under `ed/emr-creation-ed/add/pharmacy-request` and `opd-management/emr-creation-opd/add/pharmacy-request` plus a thin redirect page for the edit-mode variant.

## Findings

### Critical

- **`emr-pharmacy-request-tab-component.tsx` (both OPD and ED copies) — `isEdit` no longer requires `isEditPage`** (`src/app/(dashboard)/emr/opd/features/pharmacy-request/opd-emr-pharmacy-request-tab-component.tsx:25-31`, same in ED copy). Old guard was `isEditPage && !isDetailPage && page === "edit" && canChangeBillableRequests`; new guard drops `isEditPage &&`. On a non-EMR page that renders this component with `page=edit` in the query string, the form will mount (which can issue a pharmacy request) — most likely a regression for the create-page (no `id`) where the form would render with no `patientId`. The "create" pages just added (`opd-management/emr-creation-opd/add/pharmacy-request/page.tsx`) bypass this guard entirely by mounting `EmrPharmacyRequestForm` directly, so the redundant guard was probably removed by mistake. Restore `isEditPage &&` to the `isEdit` expression.

### Important

- **`emr-pharmacy-request-form.tsx:294` — silent no-op on missing store**. The `Select.onChange` calls `prependOpdEmrPharmacyRequestItem` only after a guard that `toast.error`s if no store is selected, but it also clears `localSearch` and resets `query` *before* that guard. After the early return, the search box is empty and the previous typing is lost. Move the `setLocalSearch("")` + `setQuery({...})` reset *after* the guard, or skip the reset entirely so the user can retry with the same search. Same block also `toast.error`s only on the missing-store case but swallows the duplicate-item case after wiping search — same complaint.

- **`emr-pharmacy-request-form.tsx:298` — duplicate detection races the form**. `opdEmrPharmacyRequestItems.some((i) => i.itemId === option.value)` checks React-Hook-Form field-array state. Field-array `prepend` is fine, but the duplicate check is also where `requestedToStoreId` is validated; if the user has added an item, then changes the requested store, no guard prevents mixing items under two different stores — the backend will reject one of them but only after submit. Worth a single-pass check before `prepend` (same root-cause guard the rest of this file uses).

- **`emr-pharmacy-request-transfer.service.ts:289-339` — DB call outside transaction**. `getOpdPharmacyTransferByRequestId(existingRequest.id)` is called *without* the `tx` client. The matching repo method (`emr-pharmacy-request.repository.ts:135-139`) was just updated to accept `tx`; the transfer service was not. This is a TOCTOU window between "transfer is PENDING inside tx" and "delete by id" — another worker could have advanced the transfer to `TRANSFERRED` in between. Pass `tx` through `getOpdPharmacyTransferByRequestId` to keep the check inside the transaction.

- **`opd/features/pharmacy-request/opd-emr-pharmacy-request-tab-component.tsx:42-55` (and ED copy) — duplicated branch in onClick**. `if (isEditPage) { ... } else { ... }` builds the same `URLSearchParams` shape in both arms with a one-line difference (`searchParams.get("tab") ?? "Pharmacy Request"`). Two-line dedupe: `const tab = isEditPage ? (searchParams.get("tab") ?? "Pharmacy Request") : "Pharmacy Request"; const params = new URLSearchParams({ tab, page: "edit", requestType: "direct" }); router.push(\`?\${params}\`);`. Same code in both the OPD and ED copies — extract to a shared helper or at least the existing `emr-pharmacy-request-mode.ts` utility.

- **`emr-pharmacy-request-form.tsx:84` — `resolveEmrPharmacyRequestType` only looks at the prop, not the URL**. The tab component computes `requestType` from `searchParams.get("requestType")` and forwards it to the form as a prop. Fine, but `resolveEmrPharmacyRequestType` defaults to `"prescription"` when `hasPrescription` is true even if no `requestType` is passed — combined with the `isEdit` regression above, this means a request page rendered without the prop will silently pick "prescription" mode (and the new "Request Pharmacy" button also defaults to `requestType=direct` explicitly, but the legacy prescription link uses `requestType=prescription` — consistent, but the implicit fallback is invisible to readers). Pass `requestType` explicitly from every call site, or drop `hasPrescription` from the resolver and require the caller to decide.

### Nit

- **`schema/pharmacy-request/create-emr-pharmacy-request.schema.ts:20` — typo fix (`staus` → `status`)**. The field is `.optional()` so the rename is safe *if* no client or server action serializes the old key. Quick grep before merge to make sure no payload still sends `staus`.

- **`emr-pharmacy-request-form.tsx:489` — `Save` → `Submit`** button copy change. Fine, but verify the existing tests / translations (`hms-docs/`) don't reference the old label.

- **`emr-pharmacy-request-form.tsx:299` — `Select` `searchable` with no item limit**. Mantine `<Select searchable>` renders the full `itemsData?.items` list as `<option>`s; with `limit: 50` you could still get 50 DOM rows per keystroke. Use Mantine `Autocomplete` (virtualized) or `Combobox` for >20 rows. 50 is borderline.

- **`emr-pharmacy-request-form.tsx:84` — both new utility files have no callers from tests outside their own unit test**. `resolveEmrPharmacyRequestType` has 2 unit tests, which is fine, but the test file is `.node.test.ts` in a path that imports `@/` aliases — verify Jest config picks those up; if Jest's `testMatch` is `**/*.test.ts` (not `*.node.test.ts`) the new tests will be silently skipped.

- **Four new `pharmacy-request/page.tsx` files are nearly identical**. `opd-management/emr-creation-opd/add/pharmacy-request/page.tsx`, `ed/emr-creation-ed/add/pharmacy-request/page.tsx`, `app/(dashboard)/emr/opd/add/pharmacy-request/page.tsx`, `app/(dashboard)/emr/ed/add/pharmacy-request/page.tsx` all wrap `EmrPharmacyRequestForm` in `PermissionGuard` with a session call. Plus the redirect-only `app/(dashboard)/emr/opd/[id]/edit/pharmacy-request/page.tsx`. Five files doing two jobs (a form-mount and a redirect). Consider one shared component imported by both root paths, or accept the duplication if the route paths must match the permission subjects exactly.

- **`emr-pharmacy-request-form.tsx:355-356` — destructured `prepend`/`remove` are conditional types**. `useFieldArray` always returns them; the conditional spread (`isDirectRequest ? removeOpdEmrPharmacyRequestItem : undefined`) downstream is fine but a reader might wonder why the destructure changed shape. One-line comment would help; or just always pass `removePharmacyRequestItem` and let the row decide.

- **package-lock.json churn (+1271/-2254 lines)** dominates the diff and includes AWS SDK upgrades (`@aws-sdk/*` 3.758 → 3.1075) and Node 18 → Node 20 engine bumps. Not a code issue but worth noting in the PR description — callers running Node 18 will break. If unrelated to this feature, consider splitting the lockfile upgrade into a separate PR so the pharmacy-request change can ship/revert independently.

## Verdict

Request changes — one regression in `isEdit` guard, one transaction-boundary bug in the cancellation service. Both are small fixes; the rest are nits. Pharmacy-request feature itself is reasonable: the `requestType` resolver + tests are a clean shape, and the cancellation hardening is a real bug fix (test coverage is good — both `EmrPharmacyRequestRepository` now accepts a `tx` and the service uses it; just the transfer-side `getOpdPharmacyTransferByRequestId` was missed).
