# Observability

The Summary Service uses structured JSON logs (pino) and Postgres audit tables. No metrics, no automated alerting in v1. This document specifies the log format, the logrotate configuration, the operator monitoring guide, and the cache-aside self-heal (no separate job — see "Redis cache self-heal" below).

---

## Log format

Every log line is a single JSON object written to stdout, with a `pino` HTTP transport re-emitting to `/var/log/ycare-summary/{api,worker}.log` and `/var/log/ycare-summary/{api,worker}.error.log`.

### Required fields

| Field | Type | Description |
|---|---|---|
| `timestamp` | string (ISO 8601) | Set by pino automatically. |
| `level` | string | One of `trace`, `debug`, `info`, `warn`, `error`, `fatal`. |
| `service` | string | `"summary-api"` or `"summary-worker"`. |
| `tenantId` | string (UUID) \| null | The verified tenant on whose behalf the work is happening. **Required** on every line that touches a request or a row. |
| `requestId` | string (UUID) \| null | Per-request correlation id, generated by the API middleware. Present on every API log line. |
| `msg` | string | Human-readable message. |

### Optional but common fields

| Field | Type | Description |
|---|---|---|
| `eventId` | string (UUID) | For worker logs, the Redis Stream event being processed. |
| `invoiceId` | string (UUID) | The CFI being touched. |
| `userId` | string (UUID) | The admin who triggered a write (status change, adjustment). |
| `durationMs` | number | Time taken for a request, a Streams event, or a DB query. |
| `route` | string | API path, e.g. `GET /consultation-fees-invoices`. |
| `statusCode` | number | HTTP response code. |
| `error.code` | string | Application error code (e.g. `VERSION_MISMATCH`). |
| `error.message` | string | Human-readable error. |
| `error.stack` | string | Stack trace (only on `error` and `fatal` levels). |

### Examples

```json
{"timestamp":"2026-01-15T08:23:11.421Z","level":"info","service":"summary-api","tenantId":"7c9e6679-7425-40de-944b-e07fc1f90ae7","requestId":"a1b2c3d4-...","route":"GET /consultation-fees-invoices","statusCode":200,"durationMs":42,"msg":"request completed"}
```

```json
{"timestamp":"2026-01-15T08:23:12.001Z","level":"info","service":"summary-worker","tenantId":"7c9e6679-7425-40de-944b-e07fc1f90ae7","eventId":"f0e1d2c3-...","invoiceId":"12345678-...","durationMs":18,"msg":"cfi.created"}
```

```json
{"timestamp":"2026-01-15T08:23:13.555Z","level":"error","service":"summary-worker","tenantId":"7c9e6679-...","eventId":"bad1bad1-...","error":{"code":"POISON_MESSAGE","message":"...","stack":"..."},"msg":"outbox.event.dead","outboxId":"deadbeef-...","attemptCount":5}
```

## Logrotate

Place this at `/etc/logrotate.d/ycare-summary`:

```
/var/log/ycare-summary/*.log {
    daily
    rotate 90
    compress
    delaycompress
    notifempty
    missingok
    create 0640 ycare-summary ycare-summary
    sharedscripts
    postrotate
        # Reopen log files. pino handles SIGUSR1 to reopen.
        /bin/systemctl kill --signal=SIGUSR1 ycare-summary-api.service 2>/dev/null || true
        /bin/systemctl kill --signal=SIGUSR1 ycare-summary-worker.service 2>/dev/null || true
    endscript
}
```

**Retention:** 90 days. Override `rotate 90` if the hospital's policy requires a different window.

**Compression:** `compress` + `delaycompress` keeps the most recent rotated file uncompressed for easy inspection; older files are gzipped.

## Logging library configuration

`pino` is configured in `src/lib/logger.ts`:

```ts
import pino from "pino";

export const logger = pino({
  level: process.env.LOG_LEVEL ?? "info",
  base: {
    service: process.env.SVC_NAME ?? "summary-unknown",  // set per-binary
  },
  timestamp: pino.stdTimeFunctions.isoTime,
  formatters: {
    level: (label) => ({ level: label }),
  },
  redact: {
    paths: ["req.headers.authorization", "req.headers['x-signature']", "*.hmacSecret"],
    censor: "[REDACTED]",
  },
});

// For request-scoped logs
export function withRequestContext(tenantId: string, requestId: string) {
  return logger.child({ tenantId, requestId });
}

// For worker event processing
export function withEventContext(tenantId: string, eventId: string) {
  return logger.child({ tenantId, eventId });
}
```

The `redact` paths ensure secrets and signature headers never land in logs, even if a developer accidentally logs a request object.

## Audit tables (not logs)

The durable audit trail for state changes is in Postgres, not in log files:

- `consultation_fees_invoice_status_changes` — every status transition (one row per change).
- `consultation_fees_invoice_adjustments` — every adjustment change (one row per change).

These tables are append-only. They are the source of truth for "who changed what when". Retention is **indefinite** (covered by the existing Postgres backup policy).

The UI's "view history" panel reads from these tables. The audit detail page links to the user who made the change via a join on `users.id`.

## Monitoring (no automated alerting in v1)

systemd's `Restart=on-failure` + `StartLimitBurst=10` + `StartLimitIntervalSec=300` bring the units back on transient crashes and stop after 10 restarts in 5 minutes on persistent failure. Beyond that, the operator monitors health with the standard tooling:

```bash
# Live tail of both units
journalctl -u ycare-summary-api -u ycare-summary-worker -f

# Errors only (for the on-call admin's morning grep)
journalctl -u ycare-summary-api -u ycare-summary-worker -p err --since "1 day ago"

# Per-day log files
tail -f /var/log/ycare-summary/api.log
tail -f /var/log/ycare-summary/worker.log
tail -f /var/log/ycare-summary/error.log
```

If the hospital later wants push-based alerts (webhook, email, pager), they can drop in `monit`, a healthchecks.io ping, a Nagios check against `/healthz`, or a custom systemd timer that greps journalctl for `level: error` and posts to a webhook. None of these are designed or built in v1 — the system already produces the right signal in structured logs.

### What to monitor (operational checklist)

| Symptom | Where to look |
|---|---|
| `summary-api` or `summary-worker` failed > 10 times in 5 min | `journalctl -u ycare-summary-* --since "5 min ago"` for the crash loop. systemd has stopped restarting; manual intervention needed. |
| `ioredis` errors repeatedly in the worker log | Redis is down or unhealthy. See `runbook.md` → "Redis is down". |
| Outbox `PENDING` count growing > 100 for more than 1 min | Worker is stuck. See `runbook.md` → "Worker is stuck". |
| Outbox `DEAD` count > 0 | A poison event is blocking; see `runbook.md` → "Outbox is accumulating DEAD events". |
| Stuck `IN_PROGRESS` rows (`locked_at` > 5 min old) not being reset by the reaper | See `runbook.md` → "Reaper is not resetting stuck IN_PROGRESS rows". |
| HMS BFF sending no events to the outbox | The HMS code change at the OPD billing insertion site is missing or broken. See `cutover-plan.md` Phase 2. |

## Outbox maintenance jobs

The worker runs three periodic jobs in addition to the main poll loop.

### Stale-claim reaper (every 5 minutes)

Resets rows that are stuck in `IN_PROGRESS` (e.g. the worker crashed mid-process):

```sql
UPDATE event_outbox
SET status = 'PENDING',
    locked_by = NULL,
    locked_at = NULL,
    last_error = coalesce(last_error, '') || ' [reaper: stale claim reset at ' || now()::text || ']'
WHERE status = 'IN_PROGRESS'
  AND locked_at < now() - interval '5 minutes';
```

The next poll re-claims the row. The CFI insert is gated by the `(tenant_id, opd_invoice_id)` UNIQUE constraint, so the re-processing is a no-op if the previous run already inserted the CFI.

### Outbox pruner (daily at 03:30)

Two passes — DONE and DEAD get different retentions:

```sql
-- DONE: short retention (7 days); enough to investigate a customer report
DELETE FROM event_outbox
WHERE status = 'DONE'
  AND completed_at < now() - interval '7 days';

-- DEAD: longer retention (30 days); operator needs time to inspect, fix the
-- root cause, and either reset to PENDING or DELETE
DELETE FROM event_outbox
WHERE status = 'DEAD'
  AND completed_at < now() - interval '30 days';
```

The DEAD retention is generous on purpose: poison messages are rare but the operator may not see the alert for a day or two, and the root-cause investigation often involves a developer who needs the row context. 30 days gives a month of slack. Tune via `OUTBOX_DONE_RETENTION_DAYS` and `OUTBOX_DEAD_RETENTION_DAYS` if the hospital's needs differ.

The table stays under 2,000 rows in steady state with these retentions.

### Redis cache self-heal (no separate job)

There is no cold-day recompute, no startup hook, and no daily cron for Redis. The cache is repaired by the read path:

- **Read flow:** API tries `HGETALL` on the day's bucket. If the key is missing, expired (TTL), or the read returns a transient error, the API falls back to a Postgres `SELECT status, COUNT(*), SUM(amount), SUM(payout_amount) FROM consultation_fees_invoices WHERE tenant_id=$1 AND billing_date::date=$2 GROUP BY status, counter_id` and writes the result back to Redis with the appropriate `EXPIRE`.
- **TTL:** every key has an expiry — `EXPIRE 86400` (24h) for the active day's buckets, `EXPIRE 604800` (7d) for past-day buckets. This forces periodic refresh.
- **Drift detection:** the `X-Cache-Status: hit | bypass` response header is the on-the-wire signal. `bypass` in the access log means Redis was down or the key was missing.

This is deliberately simpler than a daily cron rebuilding Redis. The cron would race with active writes (a 02:00 cron can clobber a 01:59:30 status change) and adds a class of "the cron ran at the wrong time" bugs. The read-path heal has no such race: the read either reads a value that was correctly written by the most recent writer, or it computes fresh from Postgres.

## What's NOT in v1

- **No metrics** (Prometheus, StatsD, etc.). The operator queries Postgres for counters (e.g., `SELECT count(*) FROM consultation_fees_invoices WHERE billing_date::date = current_date`).
- **No tracing** (OpenTelemetry, etc.). The structured logs are sufficient for the operator.
- **No remote log shipping** (Loki, ELK, etc.). Logs stay on the host.
- **No alerting SaaS** (PagerDuty, etc.). The operator uses journalctl + log files; push-based alerts are a v2 concern.

These are all reasonable v2 additions. For v1, the goal is to be operationally useful without new infrastructure.
