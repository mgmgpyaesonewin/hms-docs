# Summary Service — Design Artifacts

This folder contains the complete design for the **Summary Service** — a new Express microservice that augments the YCare HMS Next.js monolith with a `Consultation Fees Invoice` lifecycle and admin-side summary reporting.

The originating brief is at [`../summary-service-architecture-prompt.md`](../summary-service-architecture-prompt.md). Read it first for context.

## What this design covers

**v1 scope:** Auto-create a `Consultation Fees Invoice` (CFI) when an OPD invoice is generated; let admins view a summary of CFIs (filterable by date, counter, doctor, restricted search on invoice no, status) and update their status (`UNPAID` / `PAID` / `VOID`) and adjustment; cache aggregate counters in Redis for fast dashboard load.

**Deployment:** On-prem Linux server (Ubuntu 22.04 LTS) inside the hospital building, alongside the existing Next.js monolith. Two systemd units (`ycare-summary-api` and `ycare-summary-worker`) running from a single Express codebase.

**Stack:** Node.js + TypeScript + Express + Prisma (PostgreSQL) + ioredis + pino + Zod. No new cloud services. No new managed infrastructure. The new tables live in the existing HMS Postgres database; Redis is new, local-only.

## Assumed defaults (from brief Section 8.2)

The user chose to mark the remaining 7 open questions as designer defaults. The assumptions are:

| # | Question | Default |
|---|---|---|
| 1 | Denormalized display fields on CFI | **Yes** — denormalize. Drift risk is low; read speed wins. |
| 2 | RBAC permission | **New `consultation_fees:write`** permission following the existing HMS RBAC pattern. |
| 3 | Reconciliation cadence | **None — read-path cache-aside is the recovery.** No daily cron, no startup hook. The next read of a missing/stale/expired Redis key recomputes from Postgres and overwrites. (See ADR 0009.) |
| 4 | Log retention | **90 days** for app logs (logrotate); **indefinite** for DB audit tables (Postgres backup). |
| 5 | Host specs | **4 vCPU, 16 GB RAM, 100 GB SSD, gigabit ethernet, Ubuntu 22.04 LTS.** Confirm with hospital IT. |
| 6 | Backup policy | **Existing Postgres backup covers the new tables** (same DB). Confirm with hospital IT. |
| 7 | v2+ summary types | **OPD total cost, OPD by doctor, OPD by department.** Data model is summary-type-agnostic from day 1. |

Override any of these before implementation if the hospital's reality differs.

## Future work (out of scope for v1)

These items are explicitly deferred. They are constraints on the v1 data model (so the v2 implementation is non-breaking) but they do not ship in v1.

### Doctor payout workflow

The CFI's `payout_amount` answers "how much does the doctor earn for this consultation?" It does **not** answer "has the doctor been paid?" The actual money movement (cash, bank transfer, payroll batch) is a separate workflow that lives in the hospital's finance / payroll system. v1 does not build this workflow, but the v1 data model must not preclude it.

**v1 scope:** the CFI tracks what is owed to the doctor. It does not track what has been disbursed.

**v2+ scope:** a new `doctor_payouts` (or `payout_batches`) table tracks actual money movement; the CFI's `payout_amount` is referenced from the new table as a foreign key.

**Status semantics are strict:** the CFI's `status` (`UNPAID` / `PAID` / `VOID`) refers to the **patient's payment to the hospital** for the consultation fee, **not** the doctor's receipt of payout. Future payout states (e.g., `PAYABLE`, `DISBURSED`) must not be conflated with the existing CFI status.

**Constraints the v1 data model must respect:**

1. `consultation_fees_invoices.payout_amount` is a stored `NUMERIC(12,2)` column.
2. No foreign key from `consultation_fees_invoices` to a future `payout` table in v1; the link is added when the payout workflow is built.
3. The CFI's `status` enum is exactly `('UNPAID', 'PAID', 'VOID')` — do not add `PAYABLE`, `DISBURSED`, etc. to it.
4. The `payout_amount` is frozen at the moment of `PAID` transition. This is the doctor's earnings for this consultation, full stop. v2 payout workflows must treat the frozen value as authoritative.

A future implementer adding the payout workflow will add a new table, not modify the CFI table.
